# ZhikuCode vs Claude Code 功能对比分析报告（深度评估版 v8）

> **评估基准**：以 `Claude_Code源码深度架构分析.md`（598行，覆盖 Agent Loop / 工具系统 / 权限体系 / 多 Agent 协作 / System Prompt / 终端 UI / MCP 8 大模块）、原版源码目录 `/claudecode/src/`（512,685行 / 1,902文件实测）及原版**10 张架构图**（架构鸟瞰、Agent Loop 二层循环模型、queryLoop 状态转换图、Tool 接口六功能组、工具注册三种加载策略、权限判断主流程、三层协作架构、AgentTool 路由设计、SystemPromptSection 机制、MCP 四层架构）作为原版功能基准。
> 
> **v8 审查说明**：本版基于 v7 全面更新，反映了第十一章 18 项功能增强任务的全部完成。核心变更包括：① CoordinatorService runtimeEnv ConcurrentHashMap bug 修复 ② AgentMemorySnapshot + CronTaskService 原子写入（tmpFile + ATOMIC_MOVE）③ ToolRegistry getEnabledToolsSorted() prompt cache 分区排序 ④ Tool.java 三方法补全（searchHint/mapToolResult/preparePermissionMatcher）⑤ QueryEngine withheldErrors 消息扣留 + RecoveryEvent 两阶段恢复 ⑥ FileStateCache SubAgentExecutor clone/merge 集成 ⑦ SideQueryService queryStructured() + victools/jsonschema-generator 结构化输出 ⑧ McpServerEntrypoint JSON-RPC REST Controller（MCP Server 入口）⑨ PermissionRuleRepository PolicySettingsSource + PluginSettingsSource 企业策略源 ⑩ McpServerConnection listPrompts() MCP prompts/list ⑪ TeamManager + InProcessBackend Team/Swarm 框架 ⑫ ContextCollapseService 精细化折叠 + ContextCascade Level 1.5 ⑬ AgentTool team 路由（teamName→TeamManager.dispatchTasks）⑭ CoordinatorPromptBuilder MCP clients + scratchpad 增强 ⑮ MonitorTool 资源监控（CPU/Memory/Disk/JVM + RESOURCE_MONITOR 门控）⑯ MCP Streamable HTTP/WS 传输 ⑰ AgentTool Fork 路径（executeFork + getParentMessages + buildForkSystemPrompt）⑱ 编译零错误验证通过。代码规模数据均通过 wc -l 实测验证。
> 
> **评估方法**：对照架构文档中描述的每一个**功能行为**，在 ZhikuCode 源码中查找等价实现。不比较代码行数或文件数量，只看"这个功能在新版中能不能做到、做到什么程度"。
> 
> **技术栈差异声明**：原版是 TypeScript 单体（Bun + Ink 终端 UI），新版是 Java + React + Python 三层微服务架构。两者在语言特性、并发模型、UI 范式上存在本质差异，因此评估关注的是**功能等价性**而非实现路径的相似度。

## 六、架构差异影响评估

### 6.1 从终端 UI（Ink）到 Web UI（React+Vite）

| 方面 | 影响 |
|------|------|
| **功能增强** | 多窗口/标签页并行、拖放文件上传、可视化 diff 渲染、响应式布局、主题系统、设置面板、更丰富的代码高亮和 Markdown 渲染 |
| **功能损失** | Vim 键绑定（终端原生支持，Web 需额外实现）、终端硬件滚动优化（DECSTBM）、同步更新（DEC 2026 BSU/ESU）、直接进程内渲染的零延迟 |
| **净影响** | **正面** — Web UI 的可视化优势远超终端 UI 的性能优化 |

### 6.2 从 TypeScript 单体到 Java+Python+React 微服务

| 方面 | 影响 |
|------|------|
| **正面** | 更强的类型安全（Java record + sealed interface）、更成熟的 DI 框架（Spring）、更好的并发模型（Virtual Thread）、可独立扩展各层 |
| **负面** | 无法使用 AsyncGenerator 流式组合、微服务间通信增加延迟和复杂度、编译时 feature gate 用运行时条件替代（稍弱）、缺少 Bun 的极致启动速度 |
| **关键差异** | Python 服务层目前较轻量（16 文件/2,066行，tree-sitter + token 估算 + browser 服务 + CLI），与原版的单体架构差距大 |

### 6.3 性能/安全/可扩展性

| 维度 | 原版 | 新版 | 评估 |
|------|------|------|------|
| **冷启动** | Bun 极快 (<100ms) | Spring Boot (~2-5s) | 原版优 |
| **并发** | Node.js 单线程 + 异步 | Virtual Thread + 线程池 | 新版优 |
| **内存** | V8 GC | JVM G1/ZGC | 新版更可控 |
| **安全** | fail-closed 默认 | 同等设计（PermissionPipeline 7 步管线） | 持平 |

按投入产出比和业务价值排序，共 12 项核心新增功能：

#### 🔴 P0+ 级（必须立即补齐，直接影响系统核心能力）

| # | 功能名称 | 优先级 | 预估工作量 |

**① 消息扣留机制（P0+）**

- **业务价值**：当前 QueryEngine 在遇到 prompt_too_long / media_size / max_output_tokens 三类错误时直接上报给调用方（§5.1 评分 88 扣分项，§2.2 缺失 #1，图2 对齐率 82% 的主要原因）。在 SDK/API 消费场景下，调用方会过早看到中间错误并可能终止会话，而原版会先尝试内部恢复（扣留错误→尝试压缩/升级→成功则丢弃、失败才上报）
- **必要性**：这是唯一的 P0 级功能缺失（readFileState 已实现），直接影响 SDK 消费者的会话稳定性。图2 显示这是 Agent Loop 二层循环模型中两个核心差距之一
- **架构兼容性**：高度兼容。在现有 `StreamCollector.onEvent()` 中增加 `withheldMessages: List<Message>` 即可，不需要修改 QueryEngine 主循环结构。具体实现路径：
  - 收到 413 (prompt_too_long) → 加入 withheldMessages → 触发 `ContextCascade` 恢复
  - 收到 max_output_tokens → 加入 withheldMessages → 触发 escalation + recovery message 注入
  - 收到 media_size → 加入 withheldMessages → 剥离图片/大附件重试
  - 恢复成功 → 丢弃 withheldMessages；恢复失败 → 释放给 `handler.onError()`
- **预期效果**：图2 对齐率 82%→**91%**，§5.1 QueryEngine 评分 88→**92**，§9.1 加权得分 22.0→**23.0**

**② MCP Server 入口（P0+）**

- **业务价值**：当前 ZhikuCode 只是 MCP 客户端，不能作为 MCP Server 被其他 Agent 调用（图1 入口层 ❌，§2.2 缺失 #5）。这意味着在 Cursor / Windsurf / 其他多 Agent 编排场景中，ZhikuCode 只能做调用者，不能被调用，无法融入 Agent 生态工具链
- **必要性**：这是**战略级能力缺失**——Agent 生态的核心是双向可组合性（既能调用别人，也能被别人调用）。没有 MCP Server 入口，ZhikuCode 在多 Agent 时代的定位将被边缘化
- **架构兼容性**：高度兼容。现有 `ToolRegistry` + `ToolExecutionPipeline` 可完全复用，只需新增 `MCPServerEntrypoint` 类实现 stdio JSON-RPC 协议。设计方案：
  - `StdioTransport`：JSON-RPC over stdin/stdout
  - `ToolListHandler`：复用 `ToolRegistry.getActiveTools()` 暴露工具列表
  - `ToolCallHandler`：复用 `ToolExecutionPipeline.execute()` 执行工具调用
  - `ResourceListHandler`：复用现有资源列表逻辑
  - 权限策略：提供配置项（`BYPASS_PERMISSIONS` 或独立策略），外部 Agent 调用场景权限由调用方管控
  - 并发控制：复用现有 `StreamingToolExecutor` 并发机制
- **预期效果**：图1 对齐率 90%→**93%**，消除§9.2 差距 #1，增强模块评分 60→**65**

**③ resume 原子写入（P0+）**

- **业务价值**：当前 `SessionManager.saveSession()` 直接写入目标文件，崩溃时可能导致文件损坏或数据丢失（图2 ⚠️ resume 安全）。对于依赖 `/resume` 恢复长时间会话的用户，数据丢失是灾难性的
- **必要性**：数据安全底线功能，工作量极小但影响重大
- **架构兼容性**：完全兼容，仅修改 `SessionManager.saveSession()` 一个方法：
  - `Files.write(tmpPath, data)` → `Files.move(tmpPath, targetPath, StandardCopyOption.ATOMIC_MOVE)`
  - 增加 `try-finally` 确保临时文件清理
- **预期效果**：消除图2 ⚠️ resume 原子写入差距，会话可靠性从"可用"提升到"生产级"

**④ Team/Swarm 多 Agent 协作（P1）**

- **业务价值**：这是报告中**最大的单一功能缺口**（图7 对齐率仅 33%，原版 7,548行/22文件）。多 Agent 并行执行（同时修改多个文件、同时运行测试、Leader 分解任务 + Workers 并行执行）是 AI 编程助手的核心差异化能力
- **必要性**：直接决定系统在复杂项目中的效率上限。没有 Team/Swarm，面对跨模块重构、大规模代码迁移等场景只能串行处理
- **架构兼容性**：中等兼容。`SubAgentExecutor` 可作为 Worker 运行器复用，`AgentTool` 路由需扩展 teamName/name 参数。建议分三步实现：
  - 第一步：`TeamManager` + `InProcessBackend`（Virtual Thread Worker 运行器，复用 `queryEngine.execute()`）
  - 第二步：`TeamMailbox`（writeToMailbox/readMailbox）+ `SharedTaskList`（addTask/claimTask）+ Leader Permission Bridge
  - 第三步：`AgentTool.call()` 路由扩展，检测 teamName 参数时分发到 Team 路径
- **预期效果**：图7 对齐率 33%→**67-80%**，图8 对齐率 63%→**75%**，§9.2 差距 #2 消除

**⑤ Coordinator 编排模式（P1，依赖 ④）**

- **业务价值**：Coordinator 只分解任务不操作文件，Worker 汇报结果。适用于大规模任务的「总指挥→执行者」模式
- **必要性**：Team/Swarm 的高级应用模式，适用于需要全局视角协调的复杂任务（如跨 10+ 文件的 API 迁移）
- **架构兼容性**：中等兼容，依赖 Team/Swarm 基础设施。核心实现：
  - `CoordinatorToolFilter`：屏蔽 BashTool/FileEdit/FileWrite 等直接操作工具
  - `WorkerContextInjector`：为 Worker 注入任务上下文 + 限定工具范围
  - `Scratchpad`：ConcurrentHashMap<String, String> 跨 Worker 共享临时存储
  - `ResultAggregator`：汇总 Worker 结果给 Coordinator
- **预期效果**：图7 对齐率提升至 **90%+**，§9.2 差距 #2 完全消除

**⑥ Context Collapse 渐进式折叠（P1）**

- **业务价值**：当前五层压缩级联（§5.1）缺少 Context Collapse——保留消息骨架只清内容的精细折叠策略。原版在 AutoCompact 之前会先尝试 Collapse，避免不必要的昂贵 LLM 摘要调用
- **必要性**：在 AutoCompact 和 CollapseDrain 之间增加一个低成本压缩层，减少 API 调用开销
- **架构兼容性**：高度兼容。在现有 `ContextCascade` 中插入新的 Collapse 层即可：
  - `ContextCollapseService.collapseMessages(messages, retentionPolicy)`
  - 保留 user messages（反遗忘），保留 tool_use/tool_result 元数据（工具名+成功/失败），清除 assistant 消息具体内容替换为 "[collapsed: xxx字]"
  - 按消息重要性分级：user > tool_result > assistant > system
- **预期效果**：§5.1 恢复路径 6/7→**7/7**，上下文压缩层级完整度 100%

**⑦ /mcp /plan /review /rewind 命令（P1）**

- **业务价值**：报告显示命令覆盖率仅 24/86 = 28%（§七），这四个是用户高频使用的 P1 命令
- **必要性**：命令框架（`CommandRegistry`）已完善，实现成本极低（每个 50-100 行），投入产出比极高
- **架构兼容性**：完全兼容。每个命令只需实现 `Command` 接口 + 注册到 `CommandRegistry`：
  - `/mcp`：调用 `McpClientManager` API，格式化输出连接状态（列表/连接/断开/重连）
  - `/plan`：`EnterPlanModeTool` 已存在，只需注册 slash 命令入口
  - `/review`：创建 `ReviewCommand`，调用 `git diff` + LLM 生成代码审查报告
  - `/rewind`：基于 `SessionManager` 历史记录实现最近操作回退
- **预期效果**：命令覆盖率 24/86→**28/86 = 33%**

**⑧ 权限规则企业策略源（P1）**

- **业务价值**：当前仅 3 种规则来源（USER_SESSION/USER_GLOBAL/TOOL），缺企业级策略源（§5.3 权限系统评分 78 扣分项，§9.2 差距 #7）

```python
# services/browser_service.py — 核心浏览器自动化服务
import asyncio
import logging
import os
from datetime import datetime, timedelta
from typing import Optional
from playwright.async_api import async_playwright, Browser, BrowserContext, Page, Playwright

logger = logging.getLogger(__name__)

class BrowserSession:
    """单个浏览器会话 — 对应一个独立的 BrowserContext"""
    def __init__(self, context: BrowserContext, page: Page, created_at: datetime):
        self.context = context
        self.page = page
        self.created_at = created_at
        self.last_activity = created_at
        self.dialog_handler_set = False
        self._pending_dialog = None

    def touch(self):
        """更新最后活动时间"""
        self.last_activity = datetime.now()

    def is_expired(self, idle_timeout: timedelta) -> bool:
        return datetime.now() - self.last_activity > idle_timeout


class BrowserService:
    """
    浏览器自动化服务 — 管理 Playwright 实例和多会话。
    
    生命周期：
      startup()  → 启动 Playwright + 浏览器进程
      shutdown() → 关闭所有会话 + 浏览器进程 + Playwright
    
    会话管理：
      - 每个 session_id 对应独立的 BrowserContext + Page
      - 空闲超时自动清理（默认 5 分钟）
      - 最大并发会话数限制（默认 10）
    """
    
    def __init__(self):
        self._playwright: Optional[Playwright] = None
        self._browser: Optional[Browser] = None
        self._sessions: dict[str, BrowserSession] = {}
        self._lock = asyncio.Lock()
        self._cleanup_task: Optional[asyncio.Task] = None
        
        # 配置（可通过环境变量覆盖）
        self.browser_type = os.getenv("BROWSER_TYPE", "chromium")  # chromium/firefox/webkit
        self.headless = os.getenv("BROWSER_HEADLESS", "true").lower() == "true"
        self.idle_timeout = timedelta(minutes=int(os.getenv("BROWSER_IDLE_TIMEOUT_MIN", "5")))
        self.max_sessions = int(os.getenv("BROWSER_MAX_SESSIONS", "10"))
        self.default_timeout = int(os.getenv("BROWSER_DEFAULT_TIMEOUT_MS", "30000"))
    
    async def startup(self):
        """启动 Playwright 和浏览器进程"""
        self._playwright = await async_playwright().start()
        launcher = getattr(self._playwright, self.browser_type)
        self._browser = await launcher.launch(
            headless=self.headless,
            args=["--no-sandbox", "--disable-dev-shm-usage",  # 容器友好
                  "--disable-gpu", "--disable-extensions"]
        )
        # 启动定期清理任务
        self._cleanup_task = asyncio.create_task(self._periodic_cleanup())
        logger.info(f"BrowserService started: {self.browser_type}, headless={self.headless}")
    
    async def shutdown(self):
        """关闭所有资源"""
        if self._cleanup_task:
            self._cleanup_task.cancel()
        for sid in list(self._sessions.keys()):
            await self.close_session(sid)
        if self._browser:
            await self._browser.close()
        if self._playwright:
            await self._playwright.stop()
        logger.info("BrowserService shutdown complete")
    
    async def get_or_create_session(self, session_id: str) -> BrowserSession:
        """获取现有会话或创建新会话"""
        async with self._lock:
            if session_id in self._sessions:
                session = self._sessions[session_id]
                session.touch()
                return session
            
            if len(self._sessions) >= self.max_sessions:
                # 清理最久未活动的会话
                oldest_sid = min(self._sessions, key=lambda s: self._sessions[s].last_activity)
                await self._close_session_unsafe(oldest_sid)
                logger.warning(f"Session limit reached, evicted oldest: {oldest_sid}")
            
            context = await self._browser.new_context(
                viewport={"width": 1280, "height": 720},
                user_agent="ZhikuCode-Browser/1.0",
                ignore_https_errors=True,
            )
            page = await context.new_page()
            page.set_default_timeout(self.default_timeout)
            session = BrowserSession(context, page, datetime.now())
            self._sessions[session_id] = session
            logger.info(f"New browser session: {session_id} (total: {len(self._sessions)})")
            return session
    
    async def close_session(self, session_id: str) -> bool:
        async with self._lock:
            return await self._close_session_unsafe(session_id)
    
    async def _close_session_unsafe(self, session_id: str) -> bool:
        session = self._sessions.pop(session_id, None)
        if session:
            try:
                await session.context.close()
            except Exception as e:
                logger.warning(f"Error closing session {session_id}: {e}")
            return True
        return False
    
    async def _periodic_cleanup(self):
        """每 60 秒检查并清理过期会话"""
        while True:
            try:
                await asyncio.sleep(60)
                async with self._lock:
                    expired = [sid for sid, s in self._sessions.items() if s.is_expired(self.idle_timeout)]
                    for sid in expired:
                        await self._close_session_unsafe(sid)
                        logger.info(f"Expired session cleaned: {sid}")
            except asyncio.CancelledError:
                break  # 正常关闭
            except Exception as e:
                logger.error(f"Cleanup error: {e}")  # 异常不中断清理循环
    
    # ═══ 浏览器操作方法 ═══
    
    async def navigate(self, session_id: str, url: str,
                       wait_until: str = "load", timeout: int = None) -> dict:
        session = await self.get_or_create_session(session_id)
        resp = await session.page.goto(url, wait_until=wait_until,
                                        timeout=timeout or self.default_timeout)
        return {"url": session.page.url, "title": await session.page.title(),
                "status": resp.status if resp else None}
    
    async def screenshot(self, session_id: str, full_page: bool = False,
                         selector: str = None) -> dict:
        session = await self.get_or_create_session(session_id)
        import base64
        if selector:
            element = await session.page.query_selector(selector)
            if not element:
                raise ValueError(f"Element not found: {selector}")
            raw = await element.screenshot(type="png")
        else:
            raw = await session.page.screenshot(full_page=full_page, type="png")
        return {"screenshot_base64": base64.b64encode(raw).decode(), "size": len(raw)}
    
    async def click(self, session_id: str, selector: str, timeout: int = None) -> dict:
        session = await self.get_or_create_session(session_id)
        await session.page.click(selector, timeout=timeout or self.default_timeout)
        return {"clicked": selector, "url": session.page.url}
    
    async def type_text(self, session_id: str, selector: str, text: str,
                        timeout: int = None) -> dict:
        session = await self.get_or_create_session(session_id)
        await session.page.fill(selector, text, timeout=timeout or self.default_timeout)
        return {"filled": selector, "text_length": len(text)}
    
    async def evaluate(self, session_id: str, script: str) -> dict:
        session = await self.get_or_create_session(session_id)
        result = await session.page.evaluate(script)
        return {"result": str(result) if result is not None else None}
    
    async def extract_text(self, session_id: str, selector: str = None) -> dict:
        session = await self.get_or_create_session(session_id)
        if selector:
            element = await session.page.query_selector(selector)
            text = await element.inner_text() if element else ""
        else:
            text = await session.page.inner_text("body")
        # 截断过长文本（防止 token 溢出）
        max_len = 50000
        truncated = len(text) > max_len
        return {"text": text[:max_len], "length": len(text), "truncated": truncated}
    
    async def extract_html(self, session_id: str, selector: str = None) -> dict:
        session = await self.get_or_create_session(session_id)
        if selector:
            element = await session.page.query_selector(selector)
            html = await element.inner_html() if element else ""
        else:
            html = await session.page.content()
        max_len = 100000
        truncated = len(html) > max_len
        return {"html": html[:max_len], "length": len(html), "truncated": truncated}
    
    async def wait_for_selector(self, session_id: str, selector: str,
                                timeout: int = None) -> dict:
        session = await self.get_or_create_session(session_id)
        await session.page.wait_for_selector(selector,
                                              timeout=timeout or self.default_timeout)
        return {"found": selector}
    
    async def select_option(self, session_id: str, selector: str,
                            values: list[str]) -> dict:
        session = await self.get_or_create_session(session_id)
        selected = await session.page.select_option(selector, values)
        return {"selected": selected}
    
    async def handle_dialog(self, session_id: str, accept: bool,
                            text: str = None) -> dict:
        session = await self.get_or_create_session(session_id)
        # 注册对话框处理器（下一次对话框弹出时自动处理）
        async def on_dialog(dialog):
            if accept:
                await dialog.accept(text or "")
            else:
                await dialog.dismiss()
        session.page.once("dialog", on_dialog)
        return {"dialog_handler": "registered", "accept": accept}
    
    async def get_cookies(self, session_id: str) -> dict:
        session = await self.get_or_create_session(session_id)
        cookies = await session.context.cookies()
        return {"cookies": cookies}
    
    async def set_cookie(self, session_id: str, cookie: dict) -> dict:
        session = await self.get_or_create_session(session_id)
        await session.context.add_cookies([cookie])
        return {"cookie_set": cookie.get("name")}
```

```java
// tool/impl/WebBrowserTool.java — Java 端 Tool 接口实现
@Component
public class WebBrowserTool implements Tool {

    private final PythonCapabilityAwareClient pythonClient;
    private final FeatureFlagService featureFlags;
    private final ObjectMapper objectMapper;  // Jackson JSON 序列化
    private static final String CAPABILITY = "BROWSER_AUTOMATION";
    private static final Set<String> ALLOWED_ACTIONS = Set.of(
        "navigate", "screenshot", "click", "type", "evaluate",
        "extract_text", "extract_html", "wait_for", "select_option",
        "handle_dialog", "get_cookies", "set_cookie", "close_session");
    private static final Set<String> SAFE_PROTOCOLS = Set.of("http://", "https://");

    @Override public String getName() { return "WebBrowser"; }
    @Override public String getDescription() {
        return "Browser automation tool for navigating web pages, clicking elements, "
             + "filling forms, taking screenshots, and extracting content. "
             + "Supports JavaScript rendering and interactive web operations.";
    }
    @Override public boolean isEnabled() {
        return featureFlags.isEnabled("WEB_BROWSER_TOOL")
            && pythonClient.isCapabilityAvailable(CAPABILITY);
    }
    @Override public boolean shouldDefer() { return true; }
    @Override public boolean isConcurrencySafe(ToolInput input) { return false; }
    @Override public PermissionRequirement getPermissionRequirement() {
        return PermissionRequirement.REQUIRES_PERMISSION;
    }

    @Override
    public ToolResult call(ToolInput input, ToolUseContext context) {
        String action = input.getString("action");
        String sessionId = input.getString("session_id", context.sessionId());
        Map<String, Object> body = new HashMap<>(input.getRawData());
        body.put("session_id", sessionId);

        Optional<BrowserResponse> resp = pythonClient.callIfAvailable(
                CAPABILITY, "/api/browser/" + action, body, BrowserResponse.class);

        if (resp.isEmpty()) {
            return ToolResult.error(
                "Browser automation unavailable. Ensure playwright is installed: "
                + "pip install playwright && playwright install chromium");
        }
        BrowserResponse br = resp.get();
        if (!br.success()) {
            return ToolResult.error(br.errorCode() + ": " + br.errorMessage());
        }
        // 截图特殊处理
        if ("screenshot".equals(action) && br.data().containsKey("screenshot_base64")) {
            String b64 = (String) br.data().get("screenshot_base64");
            long size = ((Number) br.data().get("size")).longValue();
            return ToolResult.image(b64, "image/png", size);
        }
        try {
            return ToolResult.success(objectMapper.writeValueAsString(br.data()));
        } catch (com.fasterxml.jackson.core.JsonProcessingException e) {
            return ToolResult.error("Failed to serialize browser response: " + e.getMessage());
        }
    }

    // 注意：Python Pydantic 输出 snake_case JSON（error_code / error_message），
    // 项目 ObjectMapper 未配置 SNAKE_CASE 策略，需用 @JsonProperty 显式映射
    record BrowserResponse(
            boolean success,
            Map<String, Object> data,
            @com.fasterxml.jackson.annotation.JsonProperty("error_code") String errorCode,
            @com.fasterxml.jackson.annotation.JsonProperty("error_message") String errorMessage) {}
}
```

```java
import java.util.function.UnaryOperator;

// 在 record body 内新增：
private static final String CONTEXT_MODIFIER_KEY = "__contextModifier";

/** 携带 contextModifier 的新 ToolResult（modifier 存入 metadata） */
public ToolResult withContextModifier(UnaryOperator<ToolUseContext> modifier) {
    var newMetadata = new HashMap<>(this.metadata);
    newMetadata.put(CONTEXT_MODIFIER_KEY, modifier);
    return new ToolResult(this.content, this.isError, newMetadata);
}

/** 提取 contextModifier（可能为 null） */
@SuppressWarnings("unchecked")
public UnaryOperator<ToolUseContext> getContextModifier() {
    Object modifier = this.metadata.get(CONTEXT_MODIFIER_KEY);
    return modifier instanceof UnaryOperator ? (UnaryOperator<ToolUseContext>) modifier : null;
}

/** 返回可安全序列化的副本（去除不可序列化的 contextModifier） */
public ToolResult toSerializable() {
    if (!this.metadata.containsKey(CONTEXT_MODIFIER_KEY)) return this;
    var cleanMeta = new HashMap<>(this.metadata);
    cleanMeta.remove(CONTEXT_MODIFIER_KEY);
    return new ToolResult(this.content, this.isError, cleanMeta);
}
```

**① CoordinatorService getenv/setProperty 命名空间 bug 修复（P0+ 紧急，新增）**

- **问题描述**：`CoordinatorService.isCoordinatorMode()` L55-58 使用 `System.getenv("CLAUDE_CODE_COORDINATOR_MODE")` 读取 OS 环境变量，而 `matchSessionMode()` L91-95 使用 `System.setProperty()` / `System.clearProperty()` 操作 JVM 系统属性。Java 中 `getenv()` 和 `getProperty()` 是完全不同的命名空间，导致会话恢复模式切换**完全失效**
- **原因分析**：原版 Node.js 使用 `process.env` 是统一可读写对象，Java 无等价机制。`System.getenv()` 返回只读 Map，无法通过 `setProperty()` 影响
- **修复方案**（推荐）：统一使用会话级 `ConcurrentHashMap<String, String>` 存储运行时模式状态，初始化时回退读取 `System.getenv()`：

#### F3: 代码变更原子事务 + /undo 命令（🛡️ 安全 · ROI ★★★★☆）

#### F4: Memory LLM Rerank（🧠 AI 质量 · ROI ★★★★☆）

| 维度 | 说明 |
|------|------|
| **当前状态** | `MemorySearchEngine.java`(198行) 使用纯 **BM25 + 中文 Unigram/Bigram** 分词实现记忆搜索。核心参数：`K1=1.2`/`B=0.75`/`TITLE_BOOST=2.0`。分词器 (L104-136) 采用 CJK Unicode 范围检测 + Unigram/Bigram 双策略 + 英文标准分词 + 40 个中英停用词。`computeIDF()` (L158-168) 使用 `log((N-df+0.5)/(df+0.5)+1)` 公式，`computeBM25()` (L174-191) 标准 BM25 评分。**核心问题**：BM25 是基于词频的精确匹配算法，对**语义相似但用词不同**的查询无能为力（如查"数据持久化"无法召回"SQLite 存储"相关记忆；查"认证机制"召回不了"JWT Token 鉴权"相关记忆）。当前完全依赖 N-gram 重叠度，无语义理解能力 |
| **原版行为** | 原版记忆系统支持 LLM 嵌入向量（通过 SideQuery 生成 embedding）+ 关键词匹配的混合搜索，语义级召回准确率显著高于纯 BM25 |
| **增强方案** | **轻量 LLM Rerank 方案**（推荐，避免引入 embedding 模型复杂度）：|
| | **Step 1 — BM25 粗筛**：保持现有 `MemorySearchEngine.search()` 不变，扩大返回数量为 Top-20（原始候选集） |
| | **Step 2 — LLM Rerank 精排**：将 Top-20 结果 + 原始 query 组装为 rerank prompt，调用 `SideQueryService.queryStructured()` 返回排序结果。`SideQueryService.queryStructured()` (L98-135) 已实现 JSON Schema 自动注入 + Jackson 反序列化，复用成本为零。**注意 Java 泛型类型擦除**：`queryStructured()` 接收 `Class<T>` 参数，无法直接返回 `List<RerankResult>`（`Class<List<RerankResult>>` 不存在），需定义包装 DTO：`record RerankResponse(List<RerankResult> results) {}`，调用为 `queryStructured(prompt, content, RerankResponse.class)`。prompt 需指示 LLM 返回 `{"results": [{"index":0, "relevanceScore":8.5}, ...]}` 包装格式。**使用 fast model**（Haiku/DeepSeek-Chat）保持延迟 <500ms |
| | **Step 3 — 结果截取**：按 LLM 评分降序排列，取 Top-K（默认 5）返回给调用方 |
| | **降级策略**：LLM rerank 失败（超时/配额不足）时降级为纯 BM25 Top-K，确保功能可用 |
| **修改文件** | ① 新增 `MemoryRerankService.java` (~80行)：封装 rerank prompt 构建 + `SideQueryService.queryStructured()` 调用 + 降级逻辑 + `RerankResponse`/`RerankResult` DTO（包装类解决泛型擦除） |
| | ② `MemdirService.java`：重构 DI 结构以支持 `MemoryRerankService` 注入——推荐使用 `@Autowired` 字段注入（当前无参构造器不支持 Spring Bean 参数注入，最小改动 ~3行）+ `searchMemories()` 方法中插入 rerank 步骤——BM25 Top-20 → `MemoryRerankService.rerank(query, candidates)` → 返回精排后的 Top-K (~30行修改) |
| | ③ `application.yml`：新增 `app.memory.rerank-enabled=true` 开关 + `app.memory.rerank-model` 配置（默认 fast model） |
| **预估工作量** | 2-3 天 |
| **预期效果** | 记忆搜索从纯词频匹配升级为**语义级召回**，预期召回率提升 30-50%。用户查询"怎么配置数据库"时能正确召回"SQLite WAL 模式设置"等语义相关记忆，AI 上下文感知能力质的提升 |

### 12.5 关键结论

1. **最高 ROI 项**：F2（Rewind 实装）仅需 2-3 天，带来“文件安全网从 0 到 1”的确定性收益，建议最优先实施
2. **安全组合拳**：F2 + F3 构成完整的文件操作安全体系——F2 提供任意快照点回退，F3 在此基础上新增“单轮变更原子回退 + /undo 一键撤销”，消除用户对 AI 误操作的焦虑
3. **AI 质量提升路径**：F4（Memory Rerank）利用已有 `SideQueryService.queryStructured()` 零成本升级记忆搜索为语义级；F5（项目上下文持久化）让新会话首轮即感知项目结构，两项组合显著提升 AI 上下文理解能力
4. **架构健康投资**：F6（强类型迁移）消除消息管线中 `Map<String,Object>` 类型擦除风险，虽无直接用户可感知改善，但为后续多 Provider 适配和新内容块类型扩展奠定坚实基础
5. **模型无关性筛选**：原 F1（cache_control 管线打通）因仅服务于 Anthropic 单一 Provider、违反模型无关设计原则而移至排除区。所有保留功能均具备 Provider 无关性，可同等适用于 Anthropic/OpenAI/千问等任意模型
6. **精简聚焦策略**：从原始 17+ 项候选功能精简为 5 项实施，砍掉远程熔断（过度设计）、Worktree 工具（无使用场景）、ToolUseContext 扩展（过早优化）、cache_control（模型绑定）等 14 项 ROI 不足或违反设计原则的功能，确保每一项投入都有明确的代码级断裂点和确定性收益
7. **与 §11 的衍接**：本章 5 项规划与 §11 已完成的 18 项工作无重叠。F2 依赖 §11 已实现的 `FileHistoryService.trackEdit()` + `FileSnapshotRepository` 快照写入；F4 依赖 §11 已实现的 `SideQueryService.queryStructured()`——所有功能在已有基础设施上增量构建

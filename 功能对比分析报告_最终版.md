## 六、架构差异分析

### 6.1 架构对比

| 维度 | 原版 Claude Code | 新系统 ZhikunCode |
|------|-----------------|-------------------|

| 依赖注入 | 手动/模块化 | Spring Boot IoC |
| 包管理 | npm/bun | Maven + npm + pip |
| LLM 调用 | 直接 HTTP | LlmProvider 抽象层 |

| 原版功能 | 是否可实现 | 影响分析 |
|---------|-----------|---------|

| 桌面通知 | 需适配 | Web Notification API 替代 |
| 剪贴板操作 | 需适配 | 浏览器 Clipboard API 替代 |

### 6.3 新架构的优势

| 优势 | 说明 |
|------|------|
| **多模型支持** | LlmProvider 抽象层 + ModelRegistry 13 个内置模型，原版仅 Anthropic |
| **HTTP API 天然** | Web 架构天然提供 REST/SSE 接口，原版需额外 server/ 模块 |
| **多用户支持** | Spring Security + WebSocket session 隔离，原版单用户 |
| **Python 生态** | tree-sitter 代码分析 + FastAPI 微服务，原版无 Python 能力 |
| **部署灵活** | Docker Compose 编排，前后端可独立扩展 |
| **跨标签同步** | Zustand broadcastMiddleware 实现标签页间状态同步 |
| **Virtual Thread** | Java 21 虚拟线程，高并发工具执行性能更优 |

### 6.4 新架构的劣势

| 劣势 | 说明 |
|------|------|
| **启动时间** | JVM 冷启动比 Bun 慢 |
| **内存占用** | JVM 基础内存 ~256MB vs Bun ~50MB |
| **开发复杂度** | 三语言协作增加维护成本 |
| **延迟** | WebSocket 通信增加 ~5-10ms 延迟 |
| **终端体验** | Web UI 无法完全替代终端原生体验 |

## 九、架构优势与劣势总结

### 优势清单

1. **多 LLM Provider 支持**：LlmProviderRegistry + ModelRegistry 支持 13 个内置模型(Anthropic/OpenAI/Azure/Deepseek 等)，原版仅 Anthropic
2. **Web 多用户架构**：Spring Security + WebSocket session 隔离，支持多用户同时使用
3. **HTTP API 原生**：无需额外 server 模块，QueryController 直接提供 REST/SSE
4. **Python 生态集成**：tree-sitter 代码分析能力是原版不具备的
5. **部署灵活性**：Docker Compose 编排 + 前后端独立扩展
6. **权限管线超过原版**：PermissionPipeline 1,794 行 vs 原版 permissions.ts 1,486 行(121%)
7. **状态管理超过原版**：前后端状态系统 1,990 行 vs 原版 991 行
8. **Virtual Thread 并发**：Java 21 虚拟线程提供优于原版的并发工具执行能力

### 劣势清单

1. **总代码量仅 9.4%**：48,153 行 vs 512,664 行，功能深度普遍不足
2. **安全纵深差距最大**：BashTool 安全 3,685 行 vs 原版 24,717 行(15%)
3. **Bridge 系统薄弱**：1,541 行 vs 12,613 行(12%)，IDE 集成能力受限
4. **MCP 集成不足**：2,085 行 vs 13,992 行(15%)，生态对接能力弱
5. **Compact 系统浅层**：651 行 vs 3,960 行(16%)，长期会话记忆弱
6. **前端 UI 差距大**：8,154 行 vs 原版 ~150,000 行终端 UI 代码
7. **CLI 渲染简化**：Python 466 行 vs 原版 11,883 行，终端体验差距明显
8. **Analytics 完全缺失**：无使用分析和性能追踪

### 核心结论

1. **架构选型正确**：Java+Spring Boot/React+Vite/Python+FastAPI 的三层架构在多用户 Web 场景下有明确优势
2. **核心闭环完整**：P0 86.7% 覆盖率证明查询引擎→工具系统→权限管线→会话管理的核心闭环可用
3. **深度普遍不足**：9.4% 的代码量覆盖了 54.6% 的加权功能，说明骨架完整但血肉不足
4. **优先补齐安全+上下文**：这两个方向对 AI 智能水平影响最大，投入产出比最高
5. **合理放弃部分功能**：Analytics/Vim/Buddy/Coordinator 等可忽略，不影响核心智能水平
6. **4 周可达 ~70%**：按本报告路线图执行，预计 4 周内总体能力从 55% 提升至 70%

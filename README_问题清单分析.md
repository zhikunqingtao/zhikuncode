## 核心发现

### 1. 第十三章未发现但第十四章发现的问题
这些是集成测试的独特发现，不会在单元测试中出现：

| 问题 | 第十三章 | 第十四章 | 根因 |
|------|---------|---------|------|
| REST API 消息未持久化 | ❌ 未发现 | ❌ 失败 | REST 路径缺少 save() 调用 |
| ToolUseContext NPE | ❌ 未发现 | 🔧 修复 | Controller 初始化不完整 |
| 子代理模型 "haiku" | ❌ 未发现 | ❌ 失败 | 模型硬编码+无 Provider |
| 自动压缩阈值异常 | ❌ 未发现 | ⚠️ 异常 | 真实消息量触发缺陷 |

### 2. 硬编码问题集中在多个模块
建议统一建立配置管理体系：
- `AgentDefinition.EXPLORE/GUIDE` — 默认模型硬编码 "haiku"
- `AgentTool.getInputSchema()` — 模型枚举硬编码
- 建议创建 `ModelConfiguration.java` 集中管理

### 3. 测试基础设施不完整
- 构造器签名变更未自动同步（7 个测试文件，73 个测试）
- Mock 不完整（19 + 45 个 NPE）
- 环境初始化缺失（BashParserGoldenTest）

## 关键约定

### 问题ID 命名规则
```
P-{MODULE}-{NUMBER}

P-AG-01 = Agent Loop 第 1 个问题
P-TOOL-01 = 工具系统第 1 个问题
P-PERM-01 = 权限系统第 1 个问题
...
```

### 严重度定义
- **P0 阻塞**: 核心功能无法运行，必须立即修复
- **P1 高优**: 重要功能异常，影响用户使用，本迭代需修复
- **P2 中优**: 功能不完整或性能问题，下一迭代修复
- **P3 低优**: 代码质量或边界情况，可选修复

### 修复状态
- **未修复** = 等待执行
- **✅ 已修复** = 端到端测试前已修复
- **🔧 修复中** = 当前正在修复
- **✔️ 验证中** = 修复完成，等待测试验证

**Q: 这些问题会影响用户体验吗？**

**Q: 修复这 17 个问题需要多长时间？**

A: 4-6 周。其中 Phase 1（最关键的问题）仅需 2 周。

**Q: 修复后评分能到多少？**

A: 从当前 80.6/100 提升至 88+/100（目标是逼近原版 Claude Code 的 ~95）。

**Q: 这些问题会在哪些场景下暴露？**

A: 见详细分析.md 中每个问题的"对测试的影响"章节。

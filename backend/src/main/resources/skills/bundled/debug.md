# /debug — 系统化调试流程

从错误复现到根因定位到修复验证的完整调试闭环。确保每个Bug都经过结构化分析和验证修复。

## 触发词
- "帮我调试"
- "这个报错"
- "出了bug"
- "为什么报错"
- "debug"

## 执行流程

### 第一步：错误复现（工具：Bash）

1. 分析用户提供的错误信息，提取关键要素：
   - **错误类型**：编译错误 / 运行时异常 / 逻辑错误 / 测试失败
   - **错误消息**：完整的异常信息或错误日志
   - **触发条件**：什么操作导致了错误
2. 尝试复现错误：
   - 编译错误 → 执行构建命令（`mvn compile` / `tsc --noEmit`）
   - 运行时异常 → 执行触发命令或运行测试
   - 测试失败 → 运行指定的测试用例
3. 记录复现结果和完整输出

```bash
# Java编译错误复现
cd backend && ./mvnw compile 2>&1 | tail -50

# 前端编译错误复现
cd frontend && npx tsc --noEmit 2>&1 | tail -50

# 运行特定测试复现
cd backend && ./mvnw test -pl . -Dtest=UserServiceTest 2>&1
```

> **失败处理**：无法复现时，收集更多上下文信息（环境、版本、配置），向用户确认复现步骤。

### 第二步：收集诊断信息（工具：Bash / FileRead）

1. **日志收集**：读取最新的应用日志
   ```bash
   tail -100 log/app.log
   tail -50 log/error.log
   ```
2. **堆栈分析**：从错误输出中提取完整调用栈
3. **环境状态**：检查运行时依赖和配置
   ```bash
   java -version
   node -v
   python3 --version
   ```
4. 整理诊断信息，形成初步假设

> **失败处理**：日志文件不存在或为空时，尝试其他日志路径或提高日志级别后重新复现。

### 第三步：缩小范围（工具：Search / FileRead）

使用二分法策略定位问题代码区域：

1. 从堆栈中提取关键类名/方法名/行号
2. 使用 `Search` 定位相关源码文件
3. 确定问题所在的模块层级：
   - 控制器层（Controller）
   - 服务层（Service）
   - 数据访问层（Repository/DAO）
   - 工具类（Util）
   - 配置（Config）
4. 逐层缩小到具体方法和代码行

> **失败处理**：如果堆栈不清晰，使用 `grep` 搜索错误消息关键字定位代码位置。

### 第四步：根因分析（工具：FileRead / Search）

1. 读取问题代码段及其上下文（前后各50行）
2. 分析逻辑链：数据流向、条件分支、异常路径
3. 检查常见根因：
   - **空指针**：未检查null返回值、可选值未处理
   - **类型错误**：类型不匹配、泛型擦除
   - **并发问题**：竞态条件、死锁
   - **配置错误**：属性缺失、路径错误、端口冲突
   - **依赖问题**：版本不兼容、缺少依赖
4. 检查关联代码（调用方、被调用方、接口实现）
5. 明确输出根因结论

```
🔍 根因分析结论：
- 位置：UserService.java:125 — getUser() 方法
- 原因：数据库查询返回null时未做空值检查，直接调用 .getName()
- 影响范围：所有调用 getUser() 的上层接口
```

> **失败处理**：无法确定根因时，添加调试日志后重新复现收集更多信息。

### 第五步：制定修复方案（工具：无，纯推理）

遵循**最小化改动原则**制定修复方案：

1. 列出所有需要修改的文件和行
2. 评估修复方案的风险：
   - 修改范围（单文件/多文件）
   - 是否影响公共接口
   - 是否需要数据库变更
3. 如有多种修复方式，列出各方案优劣
4. 选择最安全的方案

### 第六步：实施修复（工具：FileEdit）

1. 按方案逐一修改目标文件
2. 每个修改都附带说明注释（如有必要）
3. 确保不引入新的代码风格问题

> **失败处理**：修改导致新的编译错误时，回退变更并调整方案。

### 第七步：验证修复（工具：Bash）

1. 重新执行触发错误的原始命令
2. 确认错误不再出现
3. 检查功能行为是否符合预期

```bash
# 重新编译
cd backend && ./mvnw compile

# 重新运行失败的测试
cd backend && ./mvnw test -Dtest=UserServiceTest

# 检查日志无新错误
tail -20 log/error.log
```

> **失败处理**：修复未生效时，回到第四步重新分析根因。

### 第八步：回归检查（工具：Bash）

1. 运行完整测试套件确认无回归
   ```bash
   # 后端完整测试
   cd backend && ./mvnw test
   
   # 前端测试
   cd frontend && npm test
   ```
2. 确认没有引入新的失败测试
3. 生成调试报告

## 输出格式

```
## 🔧 调试报告

**错误描述**：{error_description}
**根因**：{root_cause}
**修复文件**：{modified_files}

### 修复内容
| 文件 | 修改 | 说明 |
|------|------|------|
| path/to/file | 第N行 | 添加空值检查 |

### 验证结果
- 错误复现：✅ 已修复（不再出现）
- 回归测试：✅ 全部通过 / ⚠️ N个已有失败（非本次引入）

### 根因总结
{一句话总结根因和修复方式}
```

## 调试原则
- **先复现再修复**：不可复现的bug不贸然修改
- **最小化改动**：只修改必要的代码，不顺带重构
- **验证闭环**：每次修复必须经过验证确认
- **不掩盖问题**：修复根因而非症状（如不能只catch异常）

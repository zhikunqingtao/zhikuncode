# /prompt-engineering — Prompt 工程优化

将模糊、口语化的原始 prompt 改写为结构化、可被大模型稳定理解的高质量提示。聚焦"角色—任务—上下文—约束—输出格式—示例"六要素的补全与对齐，输出可直接复用的 prompt。

## 触发词
- "帮我优化一下 prompt"
- "这个提示词怎么改"
- "为什么 AI 没按我说的做"
- "prompt 不好使"
- "改写一下提示词"

## 执行流程

### 第一步：原始意图解析（工具：无，纯推理）

阅读用户提供的 `raw_prompt`，识别五个核心要素：

| 要素 | 含义 | 缺失风险 |
|------|------|----------|
| **任务动词** | 用户希望模型做什么（生成/分类/翻译/审查/总结） | 模型在多个动作间犹豫 |
| **目标对象** | 任务作用的具体内容（代码/文档/数据） | 模型臆造对象 |
| **背景上下文** | 必要的领域知识、技术栈、业务约束 | 模型用通用知识糊弄 |
| **质量标准** | 成功的判断依据（行数/格式/正确性） | 输出无法验收 |
| **输出形态** | 文本/JSON/表格/代码/Markdown | 输出形式不可预测 |

> **失败处理**：原始 prompt 完全无法解析其意图时（如只有"帮帮我"），直接询问用户："你希望我做什么具体的事？请描述任务、对象和期望结果。"

### 第二步：模式诊断（工具：无，纯推理）

对照常见反模式清单，识别原始 prompt 的具体问题：

| 反模式 | 表现 | 后果 |
|--------|------|------|
| **过度笼统** | "写得好一点"、"专业一些" | 模型输出仍然平庸 |
| **目标冲突** | 同时要求"详细"和"简短" | 模型选边站，结果不可控 |
| **缺少约束** | 不限定语言/风格/长度 | 输出冗余偏题 |
| **隐藏假设** | 默认对方知道你的项目 | 模型瞎猜上下文 |
| **示例缺失** | 只描述要求不给样例 | 模型对"好"的标准理解漂移 |
| **角色错位** | 让通用助手做专业判断 | 缺乏领域置信度 |
| **一锅端** | 一个 prompt 塞 5 件事 | 模型只完成最容易的部分 |

输出诊断结论：

```
🔍 Prompt 诊断
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
原始长度：{N} 字
检测到反模式：{反模式1}、{反模式2}
关键缺失要素：{背景上下文 / 输出格式 / ...}
预测失效模式：{模型最可能的偏离方向}
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 第三步：六要素补全（工具：无，纯推理）

按结构化模板补全 prompt 的六个组成部分：

```
1. 角色 (Role)
   你是一个 {领域专家身份}，擅长 {核心能力}。

2. 任务 (Task)
   你的任务是 {单一明确的核心动作}。

3. 上下文 (Context)
   背景：{项目/技术栈/业务场景}
   已知信息：{必要的事实和约束}

4. 约束 (Constraints)
   - 必须遵守：{硬性规则}
   - 严禁出现：{反向规则}
   - 风格偏好：{语气/长度/术语习惯}

5. 输出格式 (Output Format)
   {精确的格式描述：JSON Schema / Markdown 章节 / 表格列定义}

6. 示例 (Examples)
   输入示例：{典型输入}
   输出示例：{对应的理想输出}
```

补全原则：
- **角色尽量具体**：避免"AI 助手"，使用"具备 5 年 Java 经验的后端工程师"
- **任务唯一性**：一个 prompt 只承担一个核心目标，多目标拆分为多个 prompt
- **约束正反并陈**：既写"必须做什么"，也写"不能做什么"
- **示例 1-3 个**：少于 1 个示例时，模型靠语义猜；超过 3 个示例反而增加长度成本

> **失败处理**：用户提供的上下文不足以支撑某个要素时（如不清楚输出格式），在改写后的 prompt 中标注 `{待补充：xxx}`，提示用户填充。

### 第四步：增强技巧叠加（工具：无，纯推理）

根据任务类型，按需叠加业界公认的提示工程技巧：

| 技巧 | 适用场景 | 触发模板 |
|------|----------|----------|
| **思维链 (CoT)** | 推理/数学/多步骤决策 | "请逐步思考，先列出推理过程，再给出最终结论" |
| **自我审查** | 高质量内容/代码生成 | "完成后检查是否满足全部约束，发现问题先修正再输出" |
| **少样本 (Few-shot)** | 格式严格的输出 | 在 prompt 中嵌入 2-3 组"输入→输出"对照 |
| **角色分解** | 复杂判断 | "先以审查者视角列出关注点，再以开发者视角逐一回应" |
| **明确边界** | 防止越界发挥 | "只回答以下问题，不主动扩展其他话题" |
| **格式锚定** | 需要程序化解析 | "严格按以下 JSON Schema 输出，不添加任何说明性文字" |
| **失败兜底** | 信息不足时的处理 | "若信息不足以判断，明确回复 NEED_MORE_CONTEXT 而非编造" |

> **失败处理**：当多种技巧叠加导致 prompt 过长（超过 800 字）时，保留对当前任务收益最高的 2-3 项，移除增益边际低的技巧。

### 第五步：对照输出（工具：无，纯推理）

并列展示原始 prompt 与优化后 prompt，标注每处改动的依据：

```
📊 改写对照
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
变更点 | 原文 | 改写 | 依据
1 | "写得好一点" | "代码符合 Java 21 语法且通过 SpotBugs" | 笼统 → 可验证
2 | （无） | "你是后端工程师" | 缺角色 → 加角色
3 | （无） | "输出 Markdown 含三段落：方案/风险/验证" | 缺格式 → 加格式
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 改写依据
| 改动点 | 原始问题 | 优化策略 |
|--------|----------|----------|
| 角色 | 缺失 | 注入领域专家身份 |
| ... | ... | ... |

### 使用建议
- **温度参数**：建议 {0.0-0.3 严格 / 0.7-1.0 创造} 取决于任务类型
- **测试方法**：先用 {1-2} 个边界用例验证 prompt 行为

### 待用户确认
- {是否需要补充某个上下文要素}

```

## 优化原则
- **可验证优于主观**：所有质量要求必须可被人或代码客观判断
- **单一职责**：一个 prompt 一个目标，复合任务拆分为流水线
- **示例胜过描述**：当口头描述失效时，给一个理想样例
- **约束对称**：明确"做什么"的同时明确"不做什么"
- **不空中楼阁**：不引入用户未提供的事实，避免编造伪上下文
- **可移植性**：优化后的 prompt 应能在新对话中独立运行，不依赖隐含上下文

# /stuck — 脱困诊断

当你感觉"卡住了"——反复报错、循环修复无进展、或不知道下一步该做什么——启动系统化的脱困诊断。跳出局部视角，从全局重新评估问题，找到突破口。

## 触发词
- "卡住了"
- "修不好"
- "反复报错"
- "不知道怎么办"
- "stuck"
- "没有进展"

## 执行流程

### 第一步：状态快照（工具：Bash）

冻结当前状态，收集全面的环境信息：

1. **工作区状态**：检查当前文件变更情况
   ```bash
   git status --short
   git diff --stat
   ```
2. **最近修改**：查看最近修改的文件
   ```bash
   git log --oneline -10
   find . -name "*.java" -o -name "*.tsx" -o -name "*.py" | xargs ls -lt 2>/dev/null | head -20
   ```
3. **编译状态**：尝试编译检查当前代码健康度
   ```bash
   # 后端编译
   cd backend && ./mvnw compile -q 2>&1 | tail -20
   # 前端编译
   cd frontend && npx tsc --noEmit 2>&1 | tail -20
   ```
4. **测试状态**：运行相关测试收集当前失败情况
   ```bash
   cd backend && ./mvnw test 2>&1 | grep -E "(Tests run|FAILURE|ERROR)" | tail -20
   ```
5. **日志检查**：查看最新的错误日志
   ```bash
   tail -30 log/error.log 2>/dev/null
   tail -30 log/app.log 2>/dev/null | grep -i "error\|exception"
   ```

输出状态摘要：

```
📸 当前状态快照
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
工作区：{N}个文件已修改，{M}个未暂存
编译状态：✅ 通过 / ❌ {N}个错误
测试状态：✅ 全通过 / ❌ {N}个失败
最近改动：{最近修改的3个文件}
错误日志：{最新错误摘要}
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

> **失败处理**：某项检查命令执行失败时（如mvnw不存在），跳过该项并标注"无法检查"，继续收集其他信息。

### 第二步：历史回溯（工具：Bash / FileRead）

分析最近的操作轨迹，理解"是怎么走到这一步的"：

1. **变更历史**：回溯最近的代码改动
   ```bash
   git log --oneline --all -15
   git diff HEAD~3..HEAD --stat
   ```
2. **失败模式**：分析是否存在重复的错误模式
   - 同一个错误是否反复出现？
   - 修复A是否引发了B，修复B又引发了A？
   - 是否在同一组文件上反复修改？
3. **时间线重建**：整理关键操作的时间线
   ```
   操作时间线：
   T1: 修改了 UserService.java → 编译通过
   T2: 修改了 UserController.java → 编译失败
   T3: 回退 UserController 修改 → 编译通过但测试失败
   T4: 修改 UserService.java 试图修复 → 编译失败（当前状态）
   ```

> **失败处理**：git历史不可用时（如新项目未提交），仅基于当前文件状态和错误信息分析。

### 第三步：模式识别（工具：无，纯推理）

基于收集的信息，判断"卡住"的类型：

| 类型 | 特征 | 典型原因 |
|------|------|----------|
| **循环错误** | 修A坏B，修B坏A | 耦合过紧、接口不清晰 |
| **级联失败** | 一个错误触发一连串错误 | 底层依赖出问题 |
| **环境问题** | 代码逻辑正确但无法运行 | 版本不匹配、配置错误、端口占用 |
| **依赖缺失** | 找不到类/模块/包 | 缺少依赖、构建缓存过期 |
| **架构瓶颈** | 无论怎么改都不对 | 当前设计无法满足需求 |
| **认知盲区** | 不知道问题在哪 | 缺少关键信息或理解偏差 |

输出诊断结论：

```
🔍 诊断结论
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
卡住类型：{类型名称}
置信度：{高/中/低}
关键证据：
  - {证据1}
  - {证据2}
  - {证据3}
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

> **失败处理**：无法明确判断类型时，标注为"复合问题"，同时列出最可能的两种类型。

### 第四步：替代路径探索（工具：Search / FileRead）

基于诊断结论，提出 **3 个不同方向** 的解决方案：

1. **路径A — 最小修复**（保守路线）
   - 思路：最小化改动，只解决当前报错
   - 风险：可能只是临时解决，根因未消除
   - 操作：{具体步骤}

2. **路径B — 根因修复**（推荐路线）
   - 思路：回退到最后一个稳定状态，重新设计修复方案
   - 风险：可能丢失部分已完成的工作
   - 操作：{具体步骤}

3. **路径C — 换个方向**（突破路线）
   - 思路：绕过当前困难点，用完全不同的方式实现目标
   - 风险：需要重新思考实现方式
   - 操作：{具体步骤}

对每条路径评估：

```
| 路径 | 方向 | 成功概率 | 耗时估计 | 风险等级 |
|------|------|----------|----------|----------|
| A    | 最小修复 | 40%  | 低   | 低   |
| B    | 根因修复 | 75%  | 中   | 中   |
| C    | 换方向   | 60%  | 高   | 中   |
```

> **失败处理**：如果三条路径都评估为低成功概率，建议用户先 `git stash` 保存当前工作，回退到已知稳定版本重新开始。

### 第五步：推荐行动（工具：无，纯推理）

从三条路径中选择最佳方案，给出**具体的下一步操作**：

1. 明确推荐哪条路径及原因
2. 给出立即可执行的第一步命令或操作
3. 设定检查点：执行第一步后如何判断是否走对了
4. 设定退出条件：如果这条路也不通，何时该放弃转向其他路径

## 原则
- **先停下来再行动**：卡住时最忌讳继续盲目尝试，先全面评估再动手
- **回退是最好的进攻**：`git stash` + 回退到稳定版本往往是最快的脱困方式
- **三条路不走同一个方向**：提出的替代路径必须是本质不同的方向，不是同一思路的微调
- **设置止损点**：每条路径都有明确的"何时放弃"标准，避免沉没成本陷阱

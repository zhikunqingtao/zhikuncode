### 问题清单

#### 问题 P-AG-01
- **问题ID**: P-AG-01
- **模块**: Agent Loop / QueryEngine
- **严重度**: P1
- **问题描述**: 工具摘要缺 Haiku 异步生成机制
- **详细说明**: 当工具输出超过阈值时，当前实现直接截断处理，而非使用轻量级 LLM（如 Haiku）对长工具输出生成结构化摘要。这导致 LLM 获得的工具结果信息密度降低，可能影响后续推理质量。
- **根因分析**: 
  - 设计阶段将摘要生成作为可选项，当前实现为成本考虑选择截断策略
  - 缺少 ToolResultSummarizer 服务的异步调用机制
  - 原版 Claude Code 使用 Haiku 模型异步生成摘要，节省成本同时保证信息质量
- **相关源文件**: 
  - `/backend/src/main/java/com/aicodeassistant/engine/QueryEngine.java` (约 850-920 行，工具执行后处理区域)
  - 缺失: `ToolResultSummarizer.java` 完整实现
- **第十三章评分**: 该模块总评 80/100，此问题贡献 -5 分（评分已反映）
- **第十四章状态**: ✅ 端到端通过（在实际场景中未构成阻塞，LLM 能正常继续）
- **修复状态**: 未修复
- **对测试的影响**: 
  - 单元测试中不会暴露（因为单元测试的工具输出通常较短）
  - 端到端测试中长工具输出场景也未测试此路径
  - 影响实际应用中的工具调用链路质量

#### 问题 P-AG-02
- **问题ID**: P-AG-02
- **模块**: Agent Loop / QueryEngine
- **严重度**: P2
- **问题描述**: 缺少 newContext 更新机制
- **详细说明**: 原版在每轮循环后会生成 newContext 供下一轮使用，当前实现直接修改 messages 列表。这可能导致在需要上下文分支或回溯时缺少完整的状态快照。

  - messages 列表直接修改，无法支持上下文恢复或分支执行
- **相关源文件**: 
  - `/backend/src/main/java/com/aicodeassistant/engine/QueryEngine.java` (约 950-1000 行，循环末尾区域)
- **第十三章评分**: 该模块总评 80/100，此问题贡献 -3 分（评分已反映）

### 问题清单

#### 问题 P-CTX-01
- **问题ID**: P-CTX-01
- **模块**: 上下文管理 / REST API 持久化
- **严重度**: P1
- **问题描述**: REST API 消息未持久化到数据库
- **详细说明**: `/api/query` 和 `/api/query/conversation` 执行后不保存消息到 DB。这导致跨请求多轮对话完全断裂，用户每次调用都是新的上下文。第二、第三轮对话无法访问之前的消息历史。
- **根因分析**: 
  - QueryController 中的 REST API 端点（同步/SSE/会话）执行查询后未调用 MessageRepository.save()
  - 与 WebSocket 路径形成对比（WebSocket 路径通过 WebSocketController 正确保存消息）
  - 这是设计层面的遗漏，REST 路径被视为无状态接口但实际需要保存消息
- **相关源文件**: 
  - `/backend/src/main/java/com/aicodeassistant/controller/QueryController.java` (约 50-120 行，REST API 端点)

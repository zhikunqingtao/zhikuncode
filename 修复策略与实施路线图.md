# ZhikuCode 问题修复策略与实施路线图

## 执行摘要

基于第十三章（代码级深度审计）和第十四章（端到端集成测试）的发现，ZhikuCode 存在 **17 个需要修复的问题**，其中：

- **P0 级（阻塞）**: 1 个 — 前端权限消息处理
- **P1 级（高优先）**: 7 个 — 关键功能缺陷
- **P2 级（中优先）**: 6 个 — 功能完整性问题
- **P3 级（低优先）**: 3 个 — 代码质量或降级处理

**总体修复工时预估**：4-6 周

             return llmProviderRegistry.getHaikuModel()

**执行计划**：

1. **P-AGENT-02 修复** (3-5 天)
   - 问题：Worker 工具调用无法冒泡到 Leader
   - 方案：在 SubAgentExecutor 中自动激活 BUBBLE 模式
     ```java
     if (agentType == AgentType.TEAM_WORKER) {
         this.permissionMode = PermissionMode.BUBBLE;
     }
     ```
   - 修改 ToolExecutionPipeline：当检测到 BUBBLE 模式时，将权限请求上报而不是本地决策
   - 修改文件：
     - `SubAgentExecutor.java` (300-350 行)
     - `ToolExecutionPipeline.java` (150-200 行)
     - `PermissionPipeline.java` — 新增 BUBBLE 权限冒泡处理
   - 验证：Worker 执行工具时，权限请求通过 WebSocket 上报到 Leader

2. **P-TOOL-01 修复** (3-5 天)
   - 问题：Schema 验证为骨架实现，仅做 null 检查
   - 方案：实现完整的 JSON Schema Draft-7 验证
     ```java
     private void validateSchema(Object input, JsonSchema schema) {
         JsonSchemaValidator validator = JsonSchemaValidator.forSchema(schema);
         Set<ValidationMessage> errors = validator.validate(input);
         if (!errors.isEmpty()) {
             throw new SchemaValidationException(errors);
         }
     }
     ```
   - 添加依赖：`networknt/json-schema-validator`
   - 修改文件：`ToolExecutionPipeline.java` (80-120 行)
   - 验证：
     - 参数与 Schema 匹配时通过
     - 参数类型错误时拒绝
     - 缺少必填参数时拒绝

3. **P-SP-01 修复** (1-2 周)
   - 问题：System Prompt 仅 18.6KB，需扩充至 35KB+
   - 内容扩充方向：
     1. 详细的工具使用示例（每个工具 500-800 字的使用示例和常见错误）
     2. 边界条件处理指南（文件大小限制、递归深度、性能优化建议）
     3. 多语言代码风格规范（Python/JavaScript/Java/Go 的风格要点）
     4. 安全最佳实践详述（输入验证、路径安全、命令注入防护）
     5. 错误恢复策略说明（网络超时、LLM 速率限制、工具执行失败的恢复方案）
   - 修改文件：`SystemPromptBuilder.java`，新增段落或使用外部 prompt 文件
   - 验证：
     - System Prompt 总大小达到 35KB+
     - 端到端测试中 LLM 回复质量保持或提升
     - 边界条件处理能力增强

4. **P-CTX-02 修复** (1 天)
   - 问题：自动压缩阈值过低，仅 1 条消息即触发
   - 分析：查看 CompactService 中 `shouldAutoCompactBufferBased()` 的计算逻辑
   - 修复：调整阈值，确保至少有 5+ 条消息且总 tokens > 50% context_limit 才触发
     ```java
     private boolean shouldAutoCompactBufferBased() {
         int messageCount = messages.size();
         int totalTokens = tokenCounter.count(messages);
         int threshold = (int) (contextLimit * 0.75); // 75% 触发阈值
         
         return messageCount >= 5 && totalTokens > threshold;
     }
     ```
   - 修改文件：`CompactService.java` (200-250 行)

## 风险和缓解措施

### 风险1：Model 映射问题影响多模块
- **表现**：P-AGENT-01 和 P-AGENT-03 都涉及模型硬编码
- **缓解**：统一在配置层处理，创建 ModelConfiguration 类集中管理
- **验证**：配置变更后自动化测试覆盖所有 Agent 类型

### 风险2：REST API 消息持久化引入并发问题
- **表现**：P-CTX-01 修复后可能产生消息重复或丢失
- **缓解**：添加 @Transactional + 数据库级唯一索引
- **验证**：并发测试（10 并发会话的多轮对话）

### 风险3：权限系统变更影响现有工作流
- **表现**：P-PERM-01 和 P-AGENT-02 的权限决策更改
- **缓解**：保持向后兼容，默认权限模式不变
- **验证**：已有权限测试继续通过

### FIX-S1: TokenEncryptionService 密钥管理强化

#### FIX-S1.1: Windows 平台密钥文件权限缺失

**问题定位**: `backend/src/main/java/com/aicodeassistant/mcp/TokenEncryptionService.java` L182-L190

**现状代码**:
```java
// 设置文件权限 600（仅 POSIX 系统）
try {
    Set<PosixFilePermission> perms = PosixFilePermissions.fromString("rw-------");
    Files.setPosixFilePermissions(keyFile, perms);
    log.info("Key file permissions set to 600: {}", keyFile);
} catch (UnsupportedOperationException e) {
    log.warn("POSIX file permissions not supported on this system — " +
            "please manually secure {}", keyFile);
}
```

**问题分析**: 在 Windows 系统上，POSIX 文件权限不可用，`UnsupportedOperationException` 被捕获后仅记录了一条 `warn` 日志。这意味着在 Windows 上，密钥文件以默认权限创建，任何同机用户均可读取 AES-256 主密钥，从而解密所有 OAuth 令牌。

**攻击向量**: 在多用户 Windows 服务器上，攻击者可读取 `~/.zhiku/.master-key` 获取加密密钥，进而解密 `~/.claude/mcp-tokens/` 下的所有 OAuth 令牌。

**修复方案**:
```java
import java.nio.file.attribute.AclEntry;
import java.nio.file.attribute.AclEntryPermission;
import java.nio.file.attribute.AclEntryType;
import java.nio.file.attribute.AclFileAttributeView;
import java.nio.file.attribute.UserPrincipal;

// 替换 L182-L190
// 设置文件权限：POSIX 系统用 600，Windows 用 ACL 限制为仅当前用户
try {
    Set<PosixFilePermission> perms = PosixFilePermissions.fromString("rw-------");
    Files.setPosixFilePermissions(keyFile, perms);
    log.info("Key file permissions set to 600 (POSIX): {}", keyFile);
} catch (UnsupportedOperationException e) {
    // Windows: 使用 ACL 设置仅当前用户可访问
    try {
        AclFileAttributeView aclView = Files.getFileAttributeView(
                keyFile, AclFileAttributeView.class);
        if (aclView != null) {
            UserPrincipal owner = keyFile.getFileSystem()
                    .getUserPrincipalLookupService()
                    .lookupPrincipalByName(System.getProperty("user.name"));
            AclEntry entry = AclEntry.newBuilder()
                    .setType(AclEntryType.ALLOW)
                    .setPrincipal(owner)
                    .setPermissions(
                            AclEntryPermission.READ_DATA,
                            AclEntryPermission.WRITE_DATA,
                            AclEntryPermission.READ_ATTRIBUTES,
                            AclEntryPermission.WRITE_ATTRIBUTES)
                    .build();
            aclView.setAcl(List.of(entry));
            log.info("Key file ACL restricted to current user (Windows): {}", keyFile);
        } else {
            log.error("CRITICAL: Cannot set file permissions on this system — " +
                    "master key file {} is world-readable!", keyFile);
        }
    } catch (Exception aclEx) {
        log.error("CRITICAL: Failed to restrict key file permissions: {} — " +
                "master key file {} may be world-readable!",
                aclEx.getMessage(), keyFile);
    }
}
```

**影响范围**: 仅影响 `TokenEncryptionService.generateAndPersistKey()` 方法  
**验证方法**: 在 Windows 环境上运行，检查 `~/.zhiku/.master-key` 的 ACL 是否仅允许当前用户访问  
**数据迁移**: 不需要。已有密钥文件需手动修复权限

#### FIX-S1.2: 加密令牌缺少 HMAC 完整性验证

**问题定位**: `backend/src/main/java/com/aicodeassistant/mcp/TokenEncryptionService.java` L97-L124

**现状代码**:
```java
public String decrypt(String encrypted) {
    if (encrypted == null || encrypted.isEmpty()) {
        return encrypted;
    }
    // 向后兼容：不以 ENC: 开头的视为明文
    if (!encrypted.startsWith(ENCRYPTED_PREFIX)) {
        return encrypted;
    }

    try {
        byte[] combined = Base64.getDecoder().decode(
                encrypted.substring(ENCRYPTED_PREFIX.length()));
        // ... 直接解密，无完整性预检
    }
}
```

**问题分析**: 虽然 AES-GCM 本身提供认证加密（Authenticated Encryption），GCM tag 已内嵌在密文中，但当前实现缺少对 Base64 编码层的完整性校验。如果加密数据在存储层（文件系统）被篡改，`Base64.getDecoder().decode()` 可能产生异常的二进制数据，导致 GCM 解密失败时抛出泛化的 `RuntimeException` 而非明确的完整性校验错误。

**攻击向量**: 攻击者篡改 `~/.claude/mcp-tokens/` 下的加密文件，虽然无法解密，但可能触发不友好的异常堆栈暴露内部信息。

**修复方案**:
```java
// 替换 L97-L124 的 decrypt 方法
public String decrypt(String encrypted) {
    if (encrypted == null || encrypted.isEmpty()) {
        return encrypted;
    }
    // 向后兼容：不以 ENC: 开头的视为明文
    if (!encrypted.startsWith(ENCRYPTED_PREFIX)) {
        return encrypted;
    }

    try {
        String base64Part = encrypted.substring(ENCRYPTED_PREFIX.length());
        byte[] combined;
        try {
            combined = Base64.getDecoder().decode(base64Part);
        } catch (IllegalArgumentException e) {
            throw new SecurityException(
                    "Token data corrupted: invalid Base64 encoding", e);
        }

        if (combined.length < GCM_IV_LENGTH + 16) { // GCM tag 至少 16 bytes
            throw new SecurityException(
                    "Token data corrupted: ciphertext too short (expected >= "
                    + (GCM_IV_LENGTH + 16) + " bytes, got " + combined.length + ")");
        }

        byte[] iv = new byte[GCM_IV_LENGTH];
        System.arraycopy(combined, 0, iv, 0, GCM_IV_LENGTH);

        byte[] ciphertext = new byte[combined.length - GCM_IV_LENGTH];
        System.arraycopy(combined, GCM_IV_LENGTH, ciphertext, 0, ciphertext.length);

        Cipher cipher = Cipher.getInstance(ALGORITHM);
        cipher.init(Cipher.DECRYPT_MODE, secretKey, new GCMParameterSpec(GCM_TAG_LENGTH, iv));

        byte[] plainBytes = cipher.doFinal(ciphertext);

        // UTF-8 有效性检查
        String result = new String(plainBytes, StandardCharsets.UTF_8);
        if (!isValidUtf8(result)) {
            throw new SecurityException("Decrypted data is not valid UTF-8 — possible key mismatch");
        }
        return result;
    } catch (SecurityException e) {
        throw e; // 重新抛出已明确的安全异常
    } catch (javax.crypto.AEADBadTagException e) {
        throw new SecurityException(
                "Token decryption failed: authentication tag mismatch — " +
                "data may have been tampered with or wrong key used", e);
    } catch (Exception e) {
        throw new RuntimeException("Token decryption failed", e);
    }
}

/**
 * 校验字符串是否为合法 UTF-8（不含替换字符 U+FFFD）。
 */
private boolean isValidUtf8(String s) {
    return s.indexOf('\uFFFD') == -1;
}
```

**影响范围**: `TokenEncryptionService.decrypt()`, `McpAuthTool.storeTokens()`/加载令牌处  
**验证方法**: 单元测试中篡改 Base64 数据，验证抛出 `SecurityException` 而非 `RuntimeException`  
**数据迁移**: 不需要

**[v2 修正] 异常类型优化建议**:

> 当前方案在完整性校验失败时抛出 `SecurityException`（未检查异常），但 `SecurityException` 是 Java 安全管理器相关的异常类型，语义上不完全匹配"令牌解密失败"场景，且不利于调用方（如 `McpAuthTool`）精确处理。
>
> **建议替代方案**:
> 1. **创建自定义异常**（推荐）：`TokenDecryptionException extends RuntimeException` — 语义明确，调用方可按需 catch
> 2. **简洁封装**：统一为 `RuntimeException("Token integrity check failed", cause)` — 保持简洁但丧失类型区分能力
>
> **调用方适配**: `McpAuthTool` 在加载令牌时需要在 catch 块中处理此异常类型。当前 `McpAuthTool.storeTokens()` 和令牌加载处使用泛化的 `catch (Exception e)`，迁移为自定义异常后应改为 `catch (TokenDecryptionException e)` 以提供更精确的错误信息。

#### FIX-S2.2: sed 正则模式绕过

**问题定位**: `backend/src/main/java/com/aicodeassistant/permission/PermissionPipeline.java` L84

**现状代码**:
```java
// sed -i 就地编辑
Pattern.compile("\\bsed\\s+(-[a-zA-Z]*i|-i[a-zA-Z]*)\\s+")
```

**问题分析**: 当前正则仅匹配 `sed -i` 和 `sed -ni` 等形式，但可以被以下方式绕过：
1. `sed --in-place` — GNU sed 的长选项形式
2. `sed -e 's/foo/bar/' -i file` — `-i` 不是第一个参数
3. `command sed -i` — 通过 `command` 前缀绕过 `\b` 边界
4. `env sed -i` — 虽然 `env` 在 BARE_SHELL_PREFIXES 中，但组合场景下可能漏检

**修复方案** [v2 修正 — 正则优化为非贪心匹配]:
```java
// 替换 L84
// sed 就地编辑（覆盖 -i、--in-place、-i 在任意位置的情况）
// [v2 修正] 改用非贪心 .*? 避免贪心匹配导致的性能回溯问题；
// 在 -i 和 --in-place 前要求 \s 确保精确匹配选项边界而非子串
Pattern.compile("\\bsed\\b\\s+.*?(?:\\s-i\\b|\\s--in-place\\b)")
```

> **[v2 修正说明]**: 原方案使用贪心 `.*` 可能在长命令字符串上引发正则回溯性能问题（catastrophic backtracking）。改用非贪心 `.*?` 后：
> - 匹配效率提升：遇到第一个 `-i` 或 `--in-place` 即停止
> - 精确性增强：`\s-i` 要求 `-i` 前必须有空格，避免误匹配 `-input` 等含 `-i` 子串的参数
> - 源码参考：`PermissionPipeline.java` L84 原始正则为 `\bsed\s+(-[a-zA-Z]*i|-i[a-zA-Z]*)\s+`

**影响范围**: `CONTENT_LEVEL_ASK_PATTERNS` 正则列表  
**验证方法**: 测试以下命令全部被拦截: `sed -i 's/a/b/' f`, `sed --in-place 's/a/b/' f`, `sed -e 's/x/y/' -i f`

**问题分析**: `SessionSnapshotService` 中的 `snapshotDir.resolve(sessionId + ".json")` 直接将 `sessionId` 拼接为文件路径。如果 `sessionId` 包含 `../`，攻击者可以通过构造恶意的 sessionId（如 `../../etc/passwd`）实现路径遍历，读取或覆写任意文件。

**攻击向量**: 攻击者发送 `POST /api/sessions/../../sensitive-file/snapshot` → `snapshotDir.resolve("../../sensitive-file.json")` 跳出快照目录，可能覆写系统文件。

```java
private void validateSessionId(String sessionId) {
    if (sessionId == null || sessionId.isBlank()
            || sessionId.contains("..") || sessionId.contains("/") || sessionId.contains("\\")) {
        throw new IllegalArgumentException(
                "Invalid sessionId: must not contain path separators or traversal sequences");
    }
}
```

### FIX-Q3: McpSseTransport pendingRequests 竞态条件

**问题定位**: `backend/src/main/java/com/aicodeassistant/mcp/McpSseTransport.java` L145-L188

**现状代码**:

**对应的 SSE 事件处理**: L269

**问题分析**: 存在以下竞态条件：
1. **双重 remove**: `sendRequest()` 的 `finally` 块 (L186) 和 `handleSseEvent()` (L269) 都会调用 `pendingRequests.remove(idStr)`。如果 SSE 响应在 `future.get()` 返回后、`finally` 执行前到达，`handleSseEvent` 会 remove 成功并 complete future，然后 `finally` 再次 remove（已无效，但引入不必要的操作）。
2. **超时后响应到达**: 如果 `future.get()` 因超时抛出 `TimeoutException`，`finally` 块会 remove entry。但此时 SSE 响应可能刚好到达，`handleSseEvent` 找不到 future（已被 remove），导致响应丢失。
3. **close() 与 sendRequest() 并发**: `close()` (L303-L306) 先 forEach 再 clear，如果此时有 sendRequest 正在执行，可能在 clear 后再次 put，导致泄漏。

**修复方案** [v2 修正 — 完全重写]:

> **[v2 修正说明]**: 原方案“移除 finally 块，由各分支自行管理 remove”在异常传播场景下会引入新问题：
> 1. 如果 HTTP POST 成功但 `future.get()` 抛出非预期异常（如 `InterruptedException`），则 pending entry 泄漏
> 2. 多个 catch 分支重复 `pendingRequests.remove(idStr)` 违反 DRY 原则
> 3. `handleSseEvent` 中的 `remove` + `null` 检查不是原子操作，仍有竞态窗口
>
> 新方案策略：
> - **保留 finally 块**确保清理
> - 在响应处理中使用 `computeIfPresent()` 原子操作替代 `remove + null` 检查
> - 在超时处理中使用原子操作 cancel future
> - 补充 `close()` 方法的并发安全说明

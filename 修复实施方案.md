### P0-1: CompactService 注入 LlmProvider

**问题描述**: CompactService 当前仅注入 TokenCounter，未注入 LlmProvider，导致 3 级降级策略中的 Level 1 LLM 摘要无法执行。SPEC §3.1.5 要求通过 `COMPACT_SYSTEM_PROMPT` 模板调用 LLM 生成高质量摘要。

**涉及文件**:
| 文件 | 操作 | 说明 |
|------|------|------|
| `backend/src/main/java/com/aicodeassistant/engine/CompactService.java` | 修改 | 注入 LlmProvider，实现 Level 1 LLM 摘要 |
| `backend/src/main/java/com/aicodeassistant/llm/LlmProviderRegistry.java` | 无需修改 | 已有 `getProvider(model)`、`getFastModel()`、`getDefaultModel()` 方法 |

**当前代码状态**:
- 构造器: `public CompactService(TokenCounter tokenCounter)` — 缺少 LlmProvider
- compact() 方法 L154 注释: `// P0: 使用关键消息选择 (跳过 LLM 摘要)` — Level 1 被跳过
- 已有常量: `AUTO_COMPACT_THRESHOLD=0.85`, `COMPACT_TARGET_RATIO=0.50`, `SUMMARY_MAX_TOKENS=4096`
- 已有三区划分: `CompactionPlan(frozenMessages, compactionMessages, preservedMessages)`
- 已有 6 级 `MessagePriority` 枚举

**预估工作量**: 2h

**详细修复步骤**:

1. **注入 LlmProviderRegistry** — 修改构造器:
   > **注意**: QueryEngine 使用 `LlmProviderRegistry` (非 `LlmProvider`) 来支持多 Provider 场景。CompactService 应保持一致，注入 Registry 并通过 `getFastModel()` + `getProvider(model)` 获取 Provider。若直接注入 `LlmProvider`，在添加第二个 Provider (如 AnthropicProvider) 时 Spring 会因歧义注入失败。

   ```java
   private final LlmProviderRegistry providerRegistry;

   public CompactService(TokenCounter tokenCounter, LlmProviderRegistry providerRegistry) {
       this.tokenCounter = tokenCounter;
       this.providerRegistry = providerRegistry;
   }
   ```

2. **定义 COMPACT_SYSTEM_PROMPT 常量** (对标原版 `prompt.ts` 的 `BASE_COMPACT_PROMPT`):
   > **关键对照**: 原版 `prompt.ts` 的 `BASE_COMPACT_PROMPT` 是一个 ~80 行的 9 段结构化摘要模板，包含 `NO_TOOLS_PREAMBLE`（禁止工具调用前言）+ `DETAILED_ANALYSIS_INSTRUCTION`（`<analysis>` 思考块）+ 9 节摘要结构（Primary Request, Key Technical Concepts, Files and Code Sections, Errors and fixes, Problem Solving, All user messages, Pending Tasks, Current Work, Optional Next Step）+ `<summary>` 输出格式 + 示例。另有 `PARTIAL_COMPACT_PROMPT` 用于部分压缩场景。Java 实现应保持语义等价。

   ```java
   private static final String NO_TOOLS_PREAMBLE = """
       CRITICAL: Respond with TEXT ONLY. Do NOT call any tools.
       Tool calls will be REJECTED and will waste your only turn.
       Your entire response must be plain text: an <analysis> block followed by a <summary> block.
       """;

   private static final String COMPACT_SYSTEM_PROMPT = NO_TOOLS_PREAMBLE + """
       Your task is to create a detailed summary of the conversation so far,
       paying close attention to the user's explicit requests and your previous actions.

       Before providing your final summary, wrap your analysis in <analysis> tags.
       In your analysis process:
       1. Chronologically analyze each message. For each section identify:
          - The user's explicit requests and intents
          - Key decisions, technical concepts and code patterns
          - Specific details: file names, code snippets, function signatures, file edits
          - Errors encountered and how they were fixed
          - User feedback, especially corrections
       2. Double-check for technical accuracy and completeness.

       Your summary should include these sections in <summary> tags:
       1. Primary Request and Intent
       2. Key Technical Concepts
       3. Files and Code Sections (with code snippets)
       4. Errors and fixes
       5. Problem Solving
       6. All user messages (non tool-result)
       7. Pending Tasks
       8. Current Work (precise description with file names)
       9. Optional Next Step (only if directly related to recent work)

       Preserve all file paths, error messages, and concrete values.
       Do NOT use vague references like "the file" - use actual paths.
       """;
   ```

3. **实现 Level 1 LLM 摘要方法**:
   > **注意**: `chatSync()` 实际签名有 6 个参数: `(model, systemPrompt, userContent, maxTokens, stopSequences, timeoutMs)`。`compactionMessages` 的类型是 `List<Message>` (非 `List<Map>`)。

   ```java
   private Optional<String> generateLlmSummary(List<Message> compactionMessages, int targetTokens) {
       try {
           String conversationText = formatMessagesForSummary(compactionMessages);
           String prompt = "Target summary length: ~" + targetTokens + " tokens.\n\n" + conversationText;
           // chatSync 完整签名: (model, systemPrompt, userContent, maxTokens, stopSequences, timeoutMs)
           // LlmProviderRegistry 没有 getDefaultProvider()，需通过 model 查找
           String model = providerRegistry.getFastModel(); // 摘要优先用轻量模型
           LlmProvider provider = providerRegistry.getProvider(model);
           String summary = provider.chatSync(
               model, COMPACT_SYSTEM_PROMPT, prompt,
               SUMMARY_MAX_TOKENS, null, 30_000L);
           if (summary != null && !summary.isBlank()) {
               int summaryTokens = tokenCounter.estimateTokens(summary);
               if (summaryTokens <= SUMMARY_MAX_TOKENS) {
                   return Optional.of(summary);
               }
           }
           return Optional.empty();
       } catch (Exception e) {
           log.warn("LLM summary failed, falling back to key message selection", e);
           return Optional.empty();
       }
   }
   ```

4. **修改 compact() 降级逻辑** — 在 Level 2 关键消息选择前插入 Level 1:
   ```java
   // Level 1: LLM 摘要
   Optional<String> summary = generateLlmSummary(plan.compactionMessages(), targetTokens);
   if (summary.isPresent()) {
       return buildCompactResultWithSummary(plan, summary.get());
   }
   // Level 2: 关键消息选择 (现有逻辑)
   // Level 3: 尾部截断 (现有逻辑)
   ```

5. **实现 CompactQualityReport 三维质量验证** (对标 SPEC):
   - `infoRetention`: 检查摘要中是否保留了文件路径和操作结果
   - `coherence`: 摘要与上下文的连贯性评分
   - `tokenEfficiency`: `summaryTokens / originalTokens` 比值

**验证方法**:
- 单元测试: Mock LlmProvider，验证 Level 1 → Level 2 → Level 3 降级链路
- 集成测试: 实际调用 LLM 生成摘要，检查输出包含文件路径和操作结果
- 边界测试: LLM 超时/返回空时是否正确降级到 Level 2

**优先级理由**: CompactService 是自动上下文压缩的核心组件。没有 LLM 摘要，长对话压缩只能使用低质量的消息选择策略，直接影响多轮对话的连续性和信息保留率。

**当前签名** (LlmProvider.java):

```java
    String systemPrompt, List<Map<String, Object>> tools,
    int maxTokens, ThinkingConfig thinkingConfig, StreamChatCallback callback);
```

**预估工作量**: 4h

**详细修复步骤**:

1. **新建 MessageParam.java** — sealed interface 利用 Java 21 pattern matching:
   > **注意**: AssistantMessage 需包含 `thinkingContent` 字段以支持 extended thinking 模型。

   ```java
   public sealed interface MessageParam {
       String role();
       record UserMessage(String content) implements MessageParam {
           public String role() { return "user"; }
       }
       record AssistantMessage(
           String content,
           List<ToolUse> toolUses,
           String thinkingContent  // extended thinking 输出 (可为 null)
       ) implements MessageParam {
           public String role() { return "assistant"; }
           public AssistantMessage(String content, List<ToolUse> toolUses) {
               this(content, toolUses, null);
           }
       }
       record ToolResultMessage(String toolUseId, String content, boolean isError) implements MessageParam {
           public String role() { return "tool"; }
       }
   }
   ```

2. **新建 ToolDefinition.java**:
   ```java
   public record ToolDefinition(
       String name, String description, Map<String, Object> inputSchema
   ) {
       public Map<String, Object> toOpenAiFormat() {
           return Map.of("type", "function", "function",
               Map.of("name", name, "description", description, "parameters", inputSchema));
       }
   }
   ```

3. **新建 SystemPrompt.java** (支持分段 + 缓存控制):
   ```java
   public record SystemPrompt(List<Segment> segments) {
       public record Segment(String text, boolean cacheControl) {}
       public String toPlainText() {
           return segments.stream().map(Segment::text).collect(Collectors.joining("\n"));
       }
   }
   ```

4. **修改 LlmProvider.streamChat() 签名**:
   ```java
   void streamChat(String model, List<MessageParam> messages,
       SystemPrompt systemPrompt, List<ToolDefinition> tools,
       int maxTokens, ThinkingConfig thinkingConfig, StreamChatCallback callback);
   ```
   同时保留旧签名为 `@Deprecated default` 方法，提供过渡期:
   ```java
   @Deprecated
   default void streamChat(String model, List<Map<String, Object>> messages,
       String systemPrompt, List<Map<String, Object>> tools,
       int maxTokens, ThinkingConfig thinkingConfig, StreamChatCallback callback) {

       List<MessageParam> typedMessages = MessageParamConverter.fromMaps(messages);
       List<ToolDefinition> typedTools = ToolDefinitionConverter.fromMaps(tools);
       SystemPrompt typedPrompt = new SystemPrompt(List.of(new SystemPrompt.Segment(systemPrompt, false)));
       streamChat(model, typedMessages, typedPrompt, typedTools, maxTokens, thinkingConfig, callback);
   }
   ```

5. **修改 OpenAiCompatibleProvider** — 在 buildOpenAiRequest() 中将 MessageParam → Map 序列化:
   ```java
   private Map<String, Object> serializeMessage(MessageParam msg) {
       return switch (msg) {
           case MessageParam.UserMessage u -> Map.of("role", "user", "content", u.content());
           case MessageParam.AssistantMessage a -> /* handle toolUses */;
           case MessageParam.ToolResultMessage t -> Map.of("role", "tool", ...);

   ```

6. **修改 QueryEngine** — 构建消息列表时使用强类型而非 Map.of()

7. **修改 CompactService** — 消息分析和优先级评估使用 MessageParam 的 pattern matching

**验证方法**:
- 编译期验证: 所有使用 `List<Map<String,Object>>` 的调用点编译通过
- 单元测试: MessageParam sealed interface 的 pattern matching 覆盖所有 variant

### P0-3: OkHttp 连接池配置外部化

**问题描述**: OpenAiCompatibleProvider 中 OkHttp 连接池参数硬编码为 `ConnectionPool(5, 30, SECONDS)`。需外部化到 `application.yml` 以支持生产环境调优。

> **勘误**: 审查报告 §8.2 称"OkHttp 连接池未配置"不准确。实际代码 L83-84 已有连接池，问题是参数硬编码。

**涉及文件**:
| 文件 | 操作 | 说明 |
|------|------|------|
| `backend/src/main/java/com/aicodeassistant/llm/impl/OpenAiCompatibleProvider.java` | 修改 | 读取配置参数 |
| `backend/src/main/resources/application.yml` | 修改 | 添加连接池配置项 |

**当前代码** (OpenAiCompatibleProvider.java L83-84):
```java
this.httpClient = new OkHttpClient.Builder()
    .connectionPool(new ConnectionPool(5, 30, java.util.concurrent.TimeUnit.SECONDS))
    .connectTimeout(Duration.ofSeconds(10))
    .readTimeout(Duration.ZERO) // SSE 无读超时
    .writeTimeout(Duration.ofSeconds(10))
    .retryOnConnectionFailure(true)
    .build();
```

**预估工作量**: 1h

**详细修复步骤**:

1. **application.yml 添加配置**:
   ```yaml
   llm:
     http:
       pool:
         max-idle-connections: 5
         keep-alive-seconds: 30
       connect-timeout-seconds: 10
       write-timeout-seconds: 10
       retry-on-failure: true
   ```

2. **新建配置类 + 启用扫描** (推荐 `@ConfigurationProperties`):
   > **注意**: Spring Boot 3.x record-based `@ConfigurationProperties` 使用 `@DefaultValue` 注解设置默认值，不支持零参构造器。需在 Application 类添加 `@EnableConfigurationProperties(LlmHttpProperties.class)` 或在配置类上加 `@ConfigurationPropertiesScan`，否则 properties 不会被加载。

   ```java
   @ConfigurationProperties(prefix = "llm.http")
   public record LlmHttpProperties(
       @DefaultValue PoolProperties pool,
       @DefaultValue("10") int connectTimeoutSeconds,
       @DefaultValue("10") int writeTimeoutSeconds,
       @DefaultValue("true") boolean retryOnFailure
   ) {
       public record PoolProperties(
           @DefaultValue("5") int maxIdleConnections,
           @DefaultValue("30") int keepAliveSeconds
       ) {}
   }
   ```

3. **修改 OpenAiCompatibleProvider 构造器** — 注入 LlmHttpProperties:
   ```java
   this.httpClient = new OkHttpClient.Builder()
       .connectionPool(new ConnectionPool(
           props.pool().maxIdleConnections(),
           props.pool().keepAliveSeconds(), TimeUnit.SECONDS))
       .connectTimeout(Duration.ofSeconds(props.connectTimeoutSeconds()))
       .readTimeout(Duration.ZERO)
       .writeTimeout(Duration.ofSeconds(props.writeTimeoutSeconds()))
       .retryOnConnectionFailure(props.retryOnFailure())
       .build();
   ```

**验证方法**:
- 启动测试: 修改 yml 中参数值，通过日志确认连接池使用新参数
- 单元测试: 验证默认值回退机制 (yml 未配置时使用 record 默认值)

**优先级理由**: 生产环境需要根据并发量调整连接池参数。硬编码虽不阻塞功能，但阻塞运维调优，且修复成本极低 (1h)。

### P1-7: 2层压缩级联 Snip/MicroCompact

**问题描述**: SPEC §3.1.6 定义了多层压缩级联，当前缺少 Snip 和 MicroCompact 两层。

> **关键对照 (原版 `query.ts` L369-467 + `microCompact.ts`)**: 原版压缩管线执行顺序为: `applyToolResultBudget` (按工具结果大小预算裁剪) → `snipCompactIfNeeded` (HISTORY_SNIP 特性，操作消息历史而非单条工具结果) → `microcompactMessages` (基于 `COMPACTABLE_TOOLS` 白名单清除旧工具结果内容，含时间触发+缓存编辑两条路径) → `applyCollapsesIfNeeded` (CONTEXT_COLLAPSE) → `autocompact` (LLM 摘要)。**注意**: 原版 microCompact **不是**"移除空行/注释"，而是**清除旧的可压缩工具(FILE_READ/SHELL/GREP/GLOB/WEB_SEARCH/WEB_FETCH/FILE_EDIT/FILE_WRITE)的结果内容**，替换为 `[Old tool result content cleared]`。Java 实现应保持语义等价，即按工具类型白名单清除旧结果，而非对内容做文本处理。

**涉及文件**:
| 文件 | 操作 | 说明 |
|------|------|------|
| `backend/src/main/java/com/aicodeassistant/engine/SnipService.java` | 新建 | 工具结果超长截断 |
| `backend/src/main/java/com/aicodeassistant/engine/MicroCompactService.java` | 新建 | 微压缩 (清除旧工具结果内容) |
| `backend/src/main/java/com/aicodeassistant/engine/QueryEngine.java` | 修改 | 集成到压缩级联 |

**SPEC 定义** (§3.1.6):
- **ToolResultBudget (applyToolResultBudget)**: 对单条工具结果按大小预算裁剪 (对应原版 `query.ts` L379-394 的 `applyToolResultBudget`)，超出部分截断中间，保留首尾，插入 `[snipped N characters]` 标记
- **Snip (snipCompactIfNeeded)**: 原版为 `HISTORY_SNIP` 特性标志下的消息历史级别 snip，操作整个消息列表而非单条结果 (源文件 `snipCompact.ts` 为 feature-flagged 模块)
- **MicroCompact**: 基于 `COMPACTABLE_TOOLS` 白名单 (FILE_READ, SHELL, GREP, GLOB, WEB_SEARCH, WEB_FETCH, FILE_EDIT, FILE_WRITE)，清除旧的可压缩工具结果内容为 `[Old tool result content cleared]`。含两条路径: 时间触发 (`maybeTimeBasedMicrocompact`) 和缓存编辑 (`cachedMicrocompactPath`)

**预估工作量**: 8h

**详细修复步骤**:

1. **新建 SnipService.java**:
   ```java
   @Service
   public class SnipService {
       private static final String SNIP_MARKER = "\n[... snipped %d characters ...]\n";

       /** 对单条工具结果进行 Snip 截断 */
       public String snipIfNeeded(String content, int budgetChars) {
           if (content.length() <= budgetChars) return content;
           // 预留 marker 文本占用空间
           int markerOverhead = String.format(SNIP_MARKER, content.length()).length();
           int available = budgetChars - markerOverhead;
           if (available <= 0) return content.substring(0, budgetChars);
           int headSize = available / 2;
           int tailSize = available - headSize;
           String head = content.substring(0, headSize);
           String tail = content.substring(content.length() - tailSize);
           int snippedCount = content.length() - headSize - tailSize;
           return head + String.format(SNIP_MARKER, snippedCount) + tail;
       }

       /**
        * 遍历消息列表，对所有工具结果应用 Snip。
        * 注意: Message sealed interface 只有 UserMessage/AssistantMessage/SystemMessage，
        * 工具结果存储在 UserMessage.toolUseResult() (String) 中。
        */
       public List<Message> snipToolResults(List<Message> messages, int budgetChars) {
           return messages.stream()
               .map(msg -> {
                   if (msg instanceof Message.UserMessage user
                           && user.toolUseResult() != null) {
                       String snipped = snipIfNeeded(user.toolUseResult(), budgetChars);
                       if (!snipped.equals(user.toolUseResult())) {
                           return new Message.UserMessage(
                               user.uuid(), user.timestamp(), user.content(),
                               snipped, user.sourceToolAssistantUUID());
                       }
                   }
                   return msg;
               })
               .toList();
       }
   }
   ```

2. **新建 MicroCompactService.java** (对标原版 `microCompact.ts` 的 `COMPACTABLE_TOOLS` 白名单清除逻辑):
   ```java
   @Service
   public class MicroCompactService {
       private static final String CLEARED_MESSAGE = "[Old tool result content cleared]";

       /** 可微压缩的工具集合 (对标原版 COMPACTABLE_TOOLS) */
       private static final Set<String> COMPACTABLE_TOOLS = Set.of(
           "FileReadTool", "BashTool", "GrepTool", "GlobTool",
           "WebSearchTool", "WebFetchTool", "FileEditTool", "FileWriteTool"
       );

       /**
        * 遍历消息列表，清除旧的可压缩工具结果内容。
        * 保留最近 N 条消息的工具结果不清除 (protected tail)。
        * 注意: 工具结果在 UserMessage.toolUseResult() 中，
        *       工具名通过关联的 AssistantMessage 的 toolUse 确定。
        */
       public MicroCompactResult compactMessages(List<Message> messages, int protectedTailSize) {
           int tokensFreed = 0;
           int boundary = Math.max(0, messages.size() - protectedTailSize);
           List<Message> result = new ArrayList<>(messages.size());
           for (int i = 0; i < messages.size(); i++) {
               Message msg = messages.get(i);
               if (i < boundary && msg instanceof Message.UserMessage user
                       && user.toolUseResult() != null
                       && !user.toolUseResult().equals(CLEARED_MESSAGE)
                       && isCompactableTool(user, messages)) {
                   tokensFreed += tokenCounter.estimateTokens(user.toolUseResult());
                   result.add(new Message.UserMessage(
                       user.uuid(), user.timestamp(), user.content(),
                       CLEARED_MESSAGE, user.sourceToolAssistantUUID()));
               } else {
                   result.add(msg);
               }
           }
           return new MicroCompactResult(result, tokensFreed);
       }

       private boolean isCompactableTool(Message.UserMessage user, List<Message> messages) {
           // 通过 sourceToolAssistantUUID 查找关联的 AssistantMessage，
           // 从其 toolUse 中获取工具名，判断是否在 COMPACTABLE_TOOLS 中
           // (简化实现: 若无法确定工具名则默认可压缩)
           return true; // TODO: 实现工具名关联查找
       }

       public record MicroCompactResult(List<Message> messages, int tokensFreed) {}
   }
   ```

3. **集成到 QueryEngine 压缩级联** — 对标原版 `query.ts` L369-467 的执行顺序:

   ```java
   // Layer 2: Snip (消息历史级 snip，可选)
   ```

### P1-14: BroadcastChannel 跨 Tab 同步

**问题描述**: SPEC §8.3 要求前端 store 通过 BroadcastChannel API 实现跨 Tab 状态同步。当前所有 14 个 store 仅使用 `persist` 中间件 (localStorage)，但缺少实时跨 Tab 广播。

**涉及文件**:
| 文件 | 操作 | 说明 |
|------|------|------|
| `frontend/src/store/broadcastMiddleware.ts` | 新建 | Zustand BroadcastChannel 中间件 |
| `frontend/src/store/sessionStore.ts` | 修改 | 集成广播中间件 |
| `frontend/src/store/configStore.ts` | 修改 | 集成广播中间件 |

**预估工作量**: 2h

**详细修复步骤**:

1. **新建 broadcastMiddleware.ts**:
   > **注意**: 必须添加 `isBroadcasting` 标志防止无限循环 (Tab A 改状态 → 广播 → Tab B 收到 → 设状态 → 广播 → Tab A 收到 → 死循环)。

   ```typescript
   export const broadcastMiddleware = <T>(channelName: string) =>
     (config: StateCreator<T>): StateCreator<T> => (set, get, api) => {
       const channel = new BroadcastChannel(channelName);
       let isBroadcasting = false; // 防止无限循环

       channel.onmessage = (event) => {
         if (event.data?.type === 'STATE_UPDATE') {
           isBroadcasting = true;
           set(event.data.state, true); // replace
           isBroadcasting = false;
         }
       };
       const wrappedSet: typeof set = (...args) => {
         set(...args);
         if (!isBroadcasting) {
           channel.postMessage({ type: 'STATE_UPDATE', state: get() });
         }
       };
       return config(wrappedSet, get, api);
     };
   ```

2. **选择性集成** — 仅对需要跨 Tab 同步的 store 启用:
   - `configStore` (主题、设置)
   - `sessionStore` (会话列表)
   - 不对 `messageStore` 启用 (消息流太频繁，且消息是会话级别不需跨 Tab)

3. **添加 BroadcastChannel 可用性检测**:
   ```typescript

   ```

### P1-16: ThinkingConfig 降级通知

**问题描述**: SPEC §3.1.3 要求跨 Provider 切换时，如果目标 Provider 不支持 extended thinking，需一次性通知用户。当前 QueryEngine 直接传递 ThinkingConfig 给 Provider，未检查 Provider 是否支持。

> **关键对照 (原版 `thinking.ts`)**: 原版区分了 **两种独立检查**: `modelSupportsThinking(model)` (基础 thinking 支持) 和 `modelSupportsAdaptiveThinking(model)` (自适应 thinking 支持)。ThinkingConfig 类型为 `{ type: 'adaptive' } | { type: 'enabled'; budgetTokens: number } | { type: 'disabled' }`。Java 版应确保对 `ThinkingConfig.Adaptive` 额外检查 adaptive 支持，而非仅检查基础 thinking 支持。

**涉及文件**:
| 文件 | 操作 | 说明 |
|------|------|------|
| `backend/src/main/java/com/aicodeassistant/engine/QueryEngine.java` | 修改 | 添加降级检查和通知逻辑 |
| `backend/src/main/java/com/aicodeassistant/llm/LlmProvider.java` | 无需修改 | 已有 `supportsThinking(String model)` (L106) |

**预估工作量**: 1.5h (LlmProvider 无需修改，仅改 QueryEngine)

**详细修复步骤**:

1. **利用已有的 supportsThinking(String model) 方法**:
   > **注意**: LlmProvider L106 已有 `default boolean supportsThinking(String model)` 方法，委托给 `getModelCapabilities(model).supportsThinking()`。无需新增方法，直接使用即可。

   ```java
   // LlmProvider.java L106 — 已存在:
   default boolean supportsThinking(String model) {
       return getModelCapabilities(model).supportsThinking();
   }
   // 无需修改 LlmProvider，只需在 QueryEngine 中调用:
   provider.supportsThinking(currentModel)
   ```

2. **QueryEngine 添加降级检查** (once-only 通知):
   > **注意**: 使用已有的 `provider.supportsThinking(model)` 方法 (需传入 model 名)。

   ```java
   private final Set<String> notifiedDowngrades = ConcurrentHashMap.newKeySet();

   private ThinkingConfig resolveThinking(ThinkingConfig config, LlmProvider provider, String model) {
       if (config != null && config.requiresThinkingSupport() && !provider.supportsThinking(model)) {
           String key = "thinking-downgrade-" + provider.getProviderName() + "-" + model;
           if (notifiedDowngrades.add(key)) {
               // 通过 WebSocket 或 callback 通知前端
               log.info("Extended thinking disabled: model {} on {} does not support it", model, provider.getProviderName());
           }
           return new ThinkingConfig.Disabled();
       }
       // 原版额外区分 adaptive thinking 支持:
       // 如果 config 是 Adaptive 但 provider 不支持 adaptive，降级为 Enabled (budgetTokens)
       if (config instanceof ThinkingConfig.Adaptive && !supportsAdaptiveThinking(provider, model)) {
           log.info("Adaptive thinking not supported for {} on {}, falling back to enabled", model, provider.getProviderName());
           return new ThinkingConfig.Enabled(10000); // 默认预算
       }
       return config;
   }

   /** 检查 provider 是否支持自适应 thinking (对标原版 modelSupportsAdaptiveThinking) */
   private boolean supportsAdaptiveThinking(LlmProvider provider, String model) {
       // 原版基于 model 名称模式匹配，此处可委托给 provider.getModelCapabilities(model)
       return provider.supportsThinking(model); // 简化: 支持 thinking 即支持 adaptive
   }
   ```

3. **在 streamChat 调用前插入**:
   ```java
   ThinkingConfig resolvedThinking = resolveThinking(thinkingConfig, currentProvider, model);
   currentProvider.streamChat(model, messages, systemPrompt, tools, maxTokens, resolvedThinking, callback);
   ```

**验证方法**:
- 单元测试: 不支持 thinking 的 Provider → ThinkingConfig 自动禁用 + 通知一次
- 单元测试: 支持 thinking 的 Provider → ThinkingConfig 透传
- 单元测试: 重复调用只通知一次 (notifiedDowngrades Set)
- 单元测试: config 为 null 时不报错 (null safety)

**优先级理由**: 用户开启 extended thinking 但切换到不支持的模型时，如果没有通知，会疑惑为何响应中没有思考过程。修复成本低 (1.5h)。

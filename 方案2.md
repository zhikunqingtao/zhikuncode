### 2.1 消息渲染管线 — 工具结果分化渲染

**现状**: `frontend/src/components/message/ToolCallBlock.tsx` (202 行) 第 153-179 行的 `ToolResultRenderer` 对所有工具类型使用统一 `CodeBlock` 渲染:

**问题**: BashTool 的终端输出、FileEditTool 的 diff、GrepTool 的搜索结果都显示为相同的纯代码块，缺乏语义化展示。

**改进方案**:

#### 2.1.1 BashTool → TerminalRenderer

```
实现路径: frontend/src/components/message/renderers/TerminalRenderer.tsx

  - 可折叠的长输出 (默认显示前 50 行 + "展开全部")
  - 复制按钮 (仅复制纯文本, 去除 ANSI 码)
预期效果: 用户一眼区分终端输出与代码, 错误信息醒目标红
```

#### 2.1.2 FileEditTool → DiffView

```
实现路径: frontend/src/components/message/renderers/DiffRenderer.tsx

预期效果: 搜索结果结构化展示, 支持快速跳转
```

#### 2.1.4 渲染器路由修改

在 `ToolCallBlock.tsx` 中替换当前的 `ToolResultRenderer`:

### 2.2 流式渲染性能优化

**现状**: `frontend/src/store/messageStore.ts` (127 行) 第 41-53 行 `appendStreamDelta` 每次 delta 触发 Immer produce + React 重渲染。高频 token 流（50-100 tokens/s）导致帧率下降。

```typescript
// messageStore.ts 第 41-53 行 — 当前实现
appendStreamDelta: (delta) => set(d => {
    if (!d.streamingMessageId) {
        const msgId = crypto.randomUUID();
        d.streamingMessageId = msgId;
        d.messages.push({ uuid: msgId, type: 'assistant', ... } as Message);
    }
    d.streamingContent += delta;  // ← 每次 delta 都触发 Immer produce + 重渲染
}),
```

**改进方案**:

#### 2.2.1 Delta 批量合并

```
实现路径: frontend/src/store/messageStore.ts — appendStreamDelta
技术方案:
  - 使用 requestAnimationFrame 节流, 16ms 内多次 delta 合并为一次 state 更新
  - 累积 buffer: 在 useRef 中收集 delta, 每帧一次性 flush 到 Zustand
伪代码:
  const bufferRef = useRef('');
  const rafRef = useRef<number>();
  function appendDelta(delta: string) {
    bufferRef.current += delta;
    if (!rafRef.current) {
      rafRef.current = requestAnimationFrame(() => {
        store.setState(d => { d.streamingContent += bufferRef.current; });
        bufferRef.current = '';
        rafRef.current = undefined;
      });
    }
  }
预期效果: 渲染频率从 50-100fps 降至稳定 60fps, CPU 占用降低 40%+
```

#### 2.2.2 StreamingText 独立状态

```
问题: streamingContent 变更触发所有 MessageList 订阅者重渲染
方案: 将 streamingContent 从 Zustand 提取为 useRef + useSyncExternalStore
       仅 streaming item 组件订阅, 其他消息项不受影响
预期效果: 流式输出时, 非 streaming 的历史消息零重渲染
```

### 4.3 StreamingToolExecutor 并发安全修复

**现状**: `backend/.../tool/StreamingToolExecutor.java` (191 行) 第 104-133 行 `processQueue()` 存在并发风险。

```java
// StreamingToolExecutor.java 第 104-132 行 — 并发竞态分析
private void processQueue() {
    while (!queue.isEmpty()) {
        TrackedTool next = queue.peek();  // 第 106 行: 非原子读取
        if (!canExecute(next)) break;
        queue.poll();                     // 第 109 行: 非原子移除
        active.incrementAndGet();         // peek 和 poll 之间可被其他线程插入
        next.state = ToolState.EXECUTING;
        Thread.startVirtualThread(() -> {
            try { ... } finally {
                active.decrementAndGet();
                processQueue();           // 第 129 行: 递归调用可导致多线程同时执行
            }
        });
    }
}
```

**竞态场景**:
1. VirtualThread A 和 B 同时在 `finally` 中调用 `processQueue()`
2. 两者都 `peek()` 到同一元素 C
3. 两者都通过 `canExecute(C)` 检查
4. 两者都 `poll()` — 第二次 poll 返回下一个元素 D，但未检查 D 的并发安全性

```
修复方案 (推荐 synchronized):

  private final Object queueLock = new Object();
  
  private void processQueue() {
      synchronized (queueLock) {
          while (!queue.isEmpty()) {
              TrackedTool next = queue.peek();
              if (!canExecute(next)) break;
              queue.poll();
              active.incrementAndGet();
              next.state = ToolState.EXECUTING;
              Thread.startVirtualThread(() -> {
                  try {
                      // ... 工具执行逻辑
                  } finally {
                      active.decrementAndGet();
                      processQueue();  // 安全: synchronized 防止并发进入
                  }
              });
          }
      }
  }

补充改进:
  1. 工具级超时: 从 Tool.getMaxExecutionTimeMs() 获取, 默认 30s
     超时后 SIGTERM → 等2s → SIGKILL (对齐原版梯度终止)
  2. 结构化进度: ToolProgress record 替代 String
     { stage: "compiling" | "testing" | "analyzing", 
       percent: 0.75, 
       detail: "Running tests: 45/60 passed" }
  3. 统计指标: 记录每个工具的平均耗时/并发度/成功率
```
